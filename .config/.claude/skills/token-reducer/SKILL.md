# Token Reducer

過去**全セッション**のトークン消費を横断監査し、**構造的に繰り返す無駄**を特定して、
**ユーザーが意識しなくても再現される仕組み**に落とした改善案＋実行計画を作る。
承認を得たら適用し、後日の縦断比較で効果を検証する。

## なぜ全履歴を見るのか（設計の核）

旧 `token-reduction-review` は1セッションしか見なかったため、「その消費が構造的か偶発的か」を
判定できず提案ゼロを量産して失敗した。**複数セッションにまたがる反復こそが「構造的必然性」の証拠**になる。
このスキルは横断データを根拠に判定する。

## 自己消費を抑える進め方（scout → deep-read）

全セッション（数十ファイル・十数MB）の生ログをこの会話に読み込んではいけない。本末転倒になる。

1. **digest 取得（決定論・LLM不使用）**：
   ```bash
   python3 "${CLAUDE_SKILL_DIR}/scripts/token-audit.py"
   ```
   全セッションを集計した JSON を返す。主な項目：
   - `summary`：総セッション数。`total_output_tokens` は **メイン会話＋サブエージェント**の合算で、`main_output_tokens` と `subagent`（サブエージェント消費トークン・全体比 `share_pct`・モデル別 `by_model`）に内訳される（委任は別モデル＝多くは haiku で動くため、委任提案はこのコストと天秤にかける）。`by_project` は **正規 git リポジトリ root 単位**で集約（worktree も同一リポのサブディレクトリも本体リポへ畳む。元 cwd は `folded_paths`）。
   - `signals`：14観点に対応した検出値。各シグナルは候補セッションIDと `applies`（該当有無）を持つ。
   - `existing_mechanisms`：既存 hook / 有効プラグイン / skill 一覧 / CLAUDE.md 行数（**重複・適用済みの再提案を防ぐ**）。`skills` は各スキルの `skill_md_lines`・`has_scripts`・`script_files`・`mechanical_markers`・`scriptable_candidate`（機械的手順の記述があるのに scripts/ が無い＝観点13候補）を含む。
   - `apply_log`：過去に本スキルが適用した変更と適用日（縦断検証の基準）。**端末ローカル**（`~/.claude/token-reducer/apply_log.json`・git非追跡）に保存され、基準日は端末ごとに独立する（履歴が端末ごとに違うため）。
   - `scanned.effective_cutoff`：`--since-days`/`--since-last-apply` 適用後に実際に使われた集計開始日時（未指定なら null）。
   - オプション：`--since-days N`（直近N日）/ `--project 名`（絞り込み）/ `--split-date YYYY-MM-DD`（適用前後比較）/ `--since-last-apply`（前回適用日以降だけを集計＝新規診断用）。

   **`--split-date` と `--since-last-apply` の使い分け**：前者は**効果検証**用で全履歴を適用日の前後に分割し1セッション平均を比較する（フィルタしない）。後者は**新規診断**用で apply_log 最新日以降のセッションだけに `signals`/ランキングを絞る。2つは**併用しない**（`--since-last-apply` が before を消して検証が壊れる）。
   - **2回目以降の運用**：① `--split-date <前回適用日>` で前回施策の効果を縦断確認 → ② `--since-last-apply` で前回以降の新セッションだけ診断し新たな構造的無駄を抽出。初回は apply_log 空＝no-op。
   - 注意：`--since-last-apply` で対象セッションが少数だと「横断反復＝構造的必然性」の判定が弱くなる。母数が小さいときは無理に提案せず見送る。
2. **生ログ精読（deep-read）は診断時に全候補へ走らせず、選択後の案だけに後ろ倒す**：
   診断（段階1）は digest の `signals` だけで全 finding を軽量ランクリスト化する。**この段階では deep-read しない**。
   ユーザーが詳細化・適用する案を選んだ後（段階2）に限り、その finding の候補セッションIDを渡して
   サブエージェント（Agent）に該当 `.jsonl` を精読させ、根本原因と適切なメカニズムを設計させる。
   **メインスレッドは要約だけ受領**する。判定が要る観点（4 LSP・5 hook前処理・6 頻出指示・
   12 検証可能ゴール）は、ランクリスト段階では digest の候補有無ベースで **確証レベル B（要 deep-read）** として
   載せ概算は粗い値で出す。確証と精緻化は選択後の deep-read で行う（＝適用しない案に deep-read を払わない）。

## 評価：データ駆動 ＋ 14観点をレンズに

`signals` と `summary.by_project` を**実測消費の大きい順**に並べ、横断出現回数で構造性を確認する。
そのうえで14観点を一巡し、各観点を**見落とし防止のレンズ**として使う。

**重要**：`applies: false` の観点は**正直に「該当なし」**と書く。問題を捏造しない。
例えばこの環境では CLAUDE.md は十分短く（観点8）、モデルはほぼ Opus 一択（観点1）なので、
該当しないことが多い。無理に改善案をひねり出すのは害。

| # | 観点 | 主な根拠（digest 内） |
|---|------|----------------------|
| 1 | セッション目的に合わないモデル | `signals.model_distribution`（軽微セッションの opus 使用候補） |
| 2 | スキル目的に合わないモデル | `signals.skill_invocations` |
| 3 | CLI で済むのに MCP 使用 | `signals.mcp_usage`（`cli_replaceable` フラグ） |
| 4 | コードインテリジェンス未使用 | `signals.code_intelligence`（コードファイル多読セッション→LSP候補） |
| 5 | hook 前処理の欠如 | `signals.hook_preprocessing`（反復する大出力） |
| 6 | 頻出指示・再利用 context の skill 未移動 | `signals.repeated_instructions.clusters` |
| 7 | CLAUDE.md の workflow 指示が skill 未移動 | `existing_mechanisms.claude_md_lines` ＋本文を精読判定 |
| 8 | CLAUDE.md > 200行 | `existing_mechanisms.claude_md_lines` |
| 9 | 詳細操作の subagent 未委任 | `signals.subagent_delegation`（高コンテキストかつ Agent 0） |
| 10 | ファイル読取が最小化されていない | `signals.file_read_minimization`（大容量full読み・反復読み） |
| 11 | 複雑タスクでプランモード未使用 | `signals.plan_mode` |
| 12 | 検証可能なゴール未設定 | 候補セッションのプロンプトを精読判定 |
| 13 | スキル/サブエージェントの決定論的手続きが未スクリプト化 | `existing_mechanisms.skills`（`scriptable_candidate: true` の候補。最終判定は SKILL.md 本文を精読） |
| 14 | skill/script ソースが過大（モジュール未分割） | `signals.large_skill_sources`（`threshold_chars` 超の `*.js`/`*.py`/`*.css`。`*.min.js` 除外） |

## 改善案は必ず「仕組み」に変換する

行動アドバイス（「プランモードを使うよう心がける」）で終わらせない。それはユーザーの記憶に依存し再現しない。
各 finding を以下のいずれかの**恒久メカニズム**に落とす。`existing_mechanisms` と照合し、既存・適用済みは除外する。

- **hook**（`settings.json`）：大出力の前処理フィルタ（PreToolUse/PostToolUse）、起動時の注意喚起（SessionStart）など。
- **settings.json**：不要 MCP/プラグインの無効化、効率の良い既定の固定。
- **skill**：頻出する手順・再利用コンテキストを skill 本体やスクリプトへ昇格（その場限りの抽出を常設化）。
- **サブエージェント**：verbose な操作（ログ取得・doc fetch・広域調査）の委譲で詳細出力をメイン会話から隔離。
- **CLAUDE.md**：スリム化・skill 移譲・コンパクション保持指示など。
- **LSP プラグイン**：typed 言語で grep+複数読み → go-to-definition に置換。
- **スクリプト**：スキル/サブエージェント内の決定論的手続きを LLM から切り出してスクリプト化（観点13）。
- **モジュール分割**：過大な skill/script ソースを責務別に分割し、各 skill にモジュールマップ（責務→ファイル）を併設（観点14）。

### skill/script 開発は継続的作業 → ソースのサイズ/モジュール性も構造的レバー（観点14）

skill 開発・script 開発は一過性のプロジェクトではなく**継続的に発生する保守作業**である。この前提に立つと、
大きな単一ソース（例：1,000 行超・数 k〜十数 k tokens のファイル）は、編集のたびに全文を再読込し恒久的に
トークンを再課金する**構造的・複利的な無駄**になる（エピソード的ではない）。`signals.large_skill_sources` が
`threshold_chars` 超のソースを毎回決定論的に surface するので、ファイルサイズ/モジュール性も構造的レバーとして評価する。

- 変換先は**責務別モジュール分割 ＋ モジュールマップ**（`SKILL.md` か `lib/README.md` に「責務 → ファイル」対応表）。
  マップが無いと全モジュール読みに戻り削減が消えるため、分割とマップは常にセットにする。
- Read 横取り hook（同一ファイル再読の抑制）は edit 後の stale/正当な再読ブロックの誤爆リスクが高く、
  correctness を損なうので採らない。分割＋マップの方が安全で恒久的。
- 分割は挙動を変えてはならない。1ファイルずつ適用し都度動作確認する（破壊的変更は個別確認）。

### 決定論的手続きは必ずスクリプト化（観点13）

スキルやサブエージェントとして常設化される（＝繰り返し実行される）手続きのうち、決定論的な部分は
必ずスクリプト化し、LLM には判断・評価・自然言語の統合など非決定的な部分だけを担わせる。決定論的
処理を毎回 LLM に再実行させると起動のたびにトークンを再課金し、本来一意に定まる結果にブレや誤りが
混入するため。

- 決定論的＝同じ入力なら出力が一意（集計・パース・整形・収集・カウント・機械的変換）。スクリプトへ
  切り出し、LLM はその出力を受け取って判断する。本スキルの `token-audit.py` が全セッション集計を担うのが既存実例。
- 非決定的＝判断・評価・優先順位付け・設計・自然言語生成。ここは LLM が担う。
- 対象は反復・常設される手続きに限る。その場限りの単発手続きは保守コストが上回るためスクリプト化しない。
- `existing_mechanisms.skills` の `scriptable_candidate: true` を起点に、SKILL.md 本文を精読して
  「決定論的かつ反復」な手続きか確認し、スクリプトの作成計画（対象手順・入出力・呼び出し箇所）を作る。

行動でしか直せない観点（モデル選択・プランモード）も、可能な限り SessionStart hook での注意喚起や
statusline 表示などで**仕組みに寄せる**。寄せられないものは「**仕組み化不可・行動依存**」と明示して区別する。

## 品質ゲート（全て満たすものだけ提案）

- **構造的必然性**：複数セッションで反復している（＝横断出現回数が根拠）。単一トピック限定の偶発消費は見送る。
- **継続的削減**：毎ターン再課金される変動コンテキスト、または恒久的に効く設定変更。単発で数百トークンは対象外。
- **具体的に実行可能**：対象ファイル/行/フック/コマンドまで落とし、適用できるコード片・設定スニペットを含める。

## 効果検証は前方視点（即時再実行ではない）

過去履歴は不変なので、設定を変えて `token-audit.py` を**即時再実行しても before/after は同値**で比較にならない。
効果は今後のセッションにしか現れない。次の2段で測る：

- **(a) 提案時の静的見積り**：digest から「このパターンは N セッションで計X トークン → 除去で
  1セッションあたり約 X/N 削減」を前方推定する。
- **(b) 適用後の縦断検証**：後日（新セッションが溜まった頃）`--split-date <適用日>` で前後分割し、
  対象シグナルの1セッション平均を比較する。

「古い既出パターンを再分析しない」狙いは、**履歴を削除せず** `--since-last-apply`（分析窓のスコープ）で
達成する。履歴削除は横断監査の自滅（次回データ枯渇・縦断検証不能）・`--resume` 不可・監査証跡の不可逆喪失を
招くため行わない。古い履歴の保持期間は `settings.json` の `cleanupPeriodDays` に委ねる。

## 出力フォーマット（2段階・遅延詳細化）

診断は固定費（digest 読込＋14観点一巡）を1回だけ払って全案を軽量提示し、高コストな deep-read／
ドラフト生成は **ユーザーが選んだ案だけ** に限定する。これでこのスキル自身の消費を、適用しない案の
詳細生成という無駄なしに抑える。

```
## 段階1: 横断診断＋改善案ランクリスト（一括・固定費1回）
digest の数値を引用し、全 finding を実測消費の大きい順にランク。各行は:
順位 / 観点# / signal / N セッションで発生・概算 X トークン / 一行メカニズム案 /
確証レベル（A=digest 確定, B=要 deep-read）。
14観点を一巡し、該当なしの観点は「該当なし」と明記。
※ ここでは deep-read もドラフトコード片生成もしない。

（ユーザーがどの案を詳細化・適用するか選択 → 段階2へ）

## 段階2: 選択された案の詳細化＋適用
選ばれた finding だけ、以下を提示する:
- 原因（必要時のみ候補セッションを deep-read して確認）→ 変換先メカニズム
  （hook/settings/skill/サブエージェント/CLAUDE.md）→ 適用ファイル/設定のドラフト（コード片）
  → 静的削減見積り。
- 実行計画: 適用先（repo / dotfiles global）・適用順序・破壊的変更の有無・依存関係・トークンインパクト順。
- 検証方法: (a) 提案時の静的見積り、(b) 適用後 --split-date による縦断比較。
```

## 承認後の適用と記録

- 段階1のランクリスト提示後、**どの案を詳細化・適用するかをユーザーに確認する**（AskUserQuestion 等）。
  選ばれた案だけ段階2（deep-read＋ドラフト生成）へ進める。これが2段階フローのヒンジ。
- ユーザー承認を得てから hook / settings.json / skill / CLAUDE.md を実際に編集する。**破壊的変更は個別に確認**する。
- 適用した変更内容と適用日を **`~/.claude/token-reducer/apply_log.json`（端末ローカル・git非追跡）** に追記する（後日の縦断検証の基準）。書込前に `mkdir -p ~/.claude/token-reducer` を実行する。skill 配下（symlink で dotfiles 管理＝全端末共有）には置かない。
  形式例：`[{"date":"2026-06-06","change":"PreToolUse hook で X を要約","target":"settings.json","signal":"hook_preprocessing"}]`

## 厳守

- 価値ある提案が一つも無ければ「**特になし**」と述べる（それが正しい挙動）。
- 生トランスクリプト全体をメイン会話に読み込まない（scout → deep-read を守る）。
- 段階1で選ばれていない案に対して deep-read／full ドラフト（コード片）を生成しない（遅延詳細化を守る）。
- 適用前に必ずユーザー承認を得る。
