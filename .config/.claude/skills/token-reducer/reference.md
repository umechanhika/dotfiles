# token-reducer — 段階2（適用/検証）リファレンス

SKILL.md の段階2（選択された案の詳細化・適用・効果検証）で**必要になったときだけ**読む。
段階1（診断・ランクリスト）では読み込まない。

## 2回目以降の運用（--split-date → --since-last-apply）

初回は apply_log が空＝no-op。2回目以降は次の順で回す：

1. `--split-date <前回適用日>` で前回施策の効果を縦断確認（全履歴を適用日の前後に分割し、対象シグナルの1セッション平均を比較。フィルタはしない）。
2. `--since-last-apply` で前回適用日以降の新セッションだけに `signals`/ランキングを絞り、新たな構造的無駄を抽出（新規診断）。

**併用しない**：`--since-last-apply` は before 区間を消すため `--split-date` の効果検証が壊れる。①は効果検証、②は新規診断スコープで役割が別。母数が小さいときは「横断反復＝構造的必然性」の判定が弱くなるので無理に提案しない。

## 効果検証（前方視点・即時再実行ではない）

過去履歴は不変なので、設定を変えて `token-audit.py` を即時再実行しても before/after は同値で比較にならない。効果は今後のセッションにしか現れない。次の2段で測る：

- **(a) 提案時の静的見積り**：digest から「このパターンは N セッションで計 X トークン → 除去で1セッションあたり約 X/N 削減」を前方推定する。
- **(b) 適用後の縦断検証**：後日（新セッションが溜まった頃）`--split-date <適用日>` で前後分割し、対象シグナルの1セッション平均を比較する。

## apply_log の書式・保存先

適用した変更内容と適用日を **`~/.claude/token-reducer/apply_log.json`（端末ローカル・git 非追跡）** に追記する（後日の縦断検証の基準）。

- 書込前に `mkdir -p ~/.claude/token-reducer` を実行する。
- skill 配下（symlink で dotfiles 管理＝全端末共有）には**置かない**。基準日は端末ごとに独立する（履歴が端末ごとに違うため）。
- 形式例：

```json
[{"date":"2026-06-06","change":"PreToolUse hook で X を要約","target":"settings.json","signal":"hook_preprocessing"}]
```
