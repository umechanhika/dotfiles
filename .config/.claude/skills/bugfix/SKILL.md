# Bugfix

不具合を修正するときは、**必ず原因を特定したうえで対策を検討する**。原因不明のまま対策すると、
その対策で本当に問題が解消するか判断できず、問題の規模に対して過剰な対応をしてしまうリスクが
あるため。以下の STEP を順に踏む。

## STEP1 原因を特定する

- 調査だけで原因特定が難しい場合は、ログを仕込んで調査する、ユーザーに協力を仰ぐなど、原因特定に
  必要な行動を検討する。
- そうした対応も難しい事象の場合は、その旨を明示したうえで原因を推測し、**「推測した原因に対する
  対策である」ことを明示する**。

## STEP2 全パターンと影響範囲を把握する

原因を特定して対策を検討する際は、想定される全パターンと、修正が他に与える影響範囲をすべて把握する。
把握漏れのある対策は、デグレ（既存挙動の退行）を生んでも気づけないため。

- 影響範囲は、修正対象の**呼び出し箇所・参照箇所を実際に洗い出して確認**し、入力や状態の取りうる
  パターン・境界値も列挙する。

## STEP3 デグレを起こさないことを検証する

デグレを起こさないことの確認は**推論だけに頼らない**。

- 既存テストの実行、再現テストおよび周辺パターンのテスト追加で裏付ける。
- テストがない／書けない箇所は**「未検証」であることを明示する**。
- アプリを実行して修正が機能するか確かめたいときは `/verify`、差分をバグ観点でレビューさせたいときは
  `/code-review` を検証手段として併用してよい。

## STEP4 根本対策で変更を最小にする

- 対策は根本原因に対する**根本対策**とする。症状だけを抑える場当たり的な対症療法（ローカルなパッチ）は、
  設計をツギハギにして技術的負債やデグレにつながるため選ばない。
- そのうえで、根本的・全体的な設計まで含めて検討し、根本対策を満たす範囲で変更を最小にする。
  根本対策に設計レベルの変更が必要なら、それを含めた範囲が最小単位であり、削るのは過小対応（対応漏れ）。
  逆に、原因と無関係な変更・先回りの一般化・ついでのリファクタなど根本対策に不要な上乗せはしない
  （＝「過剰な対応をしない」の意味）。

## STEP5 確認できない範囲はユーザーに判断を仰ぐ

- 全影響の把握やデグレ無しの確認が難しい場合は、確認できた範囲とできなかった範囲（残存リスク）を明示し、
  ユーザーに判断を仰ぐ。
