# Bugfix

不具合を修正するときは、**必ず原因を特定したうえで対策を検討する**。原因不明のまま対策すると、
その対策で本当に問題が解消するか判断できず、問題の規模に対して過剰な対応をしてしまうリスクが
あるため。以下の STEP を順に踏む。

各 STEP は「やったつもり」で終えず、**検証可能な証跡（artifact）**で裏づける（各 STEP の「### 検証」）。
検証の背骨は **症状を再現する失敗テスト（赤）→ 修正後に成功（緑）→ 既存スイートも緑**。この1本が
原因の実在（赤で再現）・修正の有効性（緑）・無デグレ（スイート緑）を同時に裏づける。
**検証の深さは重大度・影響範囲に比例**させ、軽微で局所的な不具合に重い一式を強制しない。
ただし簡略化した箇所は必ず「未検証」と明示する（簡略化＝隠蔽ではない）。

## STEP1 原因を特定する

- 調査だけで原因特定が難しい場合は、ログを仕込んで調査する、ユーザーに協力を仰ぐなど、原因特定に
  必要な行動を検討する。
- そうした対応も難しい事象の場合は、その旨を明示したうえで原因を推測し、**「推測した原因に対する
  対策である」ことを明示する**。

### 検証
- 証跡：症状を意図的に再現する**失敗テスト（赤）**か、原因行を指すログ/スタックトレース。
- 赤を作れたこと自体が「原因が実在する」証拠。再現も特定もできないときは対策に「推測」ラベルを付ける。

## STEP2 全パターンと影響範囲を把握する

原因を特定して対策を検討する際は、想定される全パターンと、修正が他に与える影響範囲をすべて把握する。
把握漏れのある対策は、デグレ（既存挙動の退行）を生んでも気づけないため。

- 影響範囲は、修正対象の**呼び出し箇所・参照箇所を実際に洗い出して確認**し、入力や状態の取りうる
  パターン・境界値も列挙する。
- 呼び出し箇所・参照箇所の洗い出しは決定論スクリプトで全列挙する（手 grep の漏れを防ぐ）：

  ```bash
  bash "${CLAUDE_SKILL_DIR}/scripts/impact-scan.sh" <修正予定シンボル>...
  ```

  出力の `## references` は **textual 一致の下限**。動的ディスパッチ/DI/リフレクション/文字列キー/
  別言語境界は拾えないので、ここを起点にコードを読んで上限側を補う。

### 検証
- 証跡：impact-scan が挙げた**全参照を1件ずつ消し込む**（各行「修正済み」か「影響なし＋理由」）。
  黙殺（言及なしのスキップ）はしない。
- 入力/状態のパターン・境界値も列挙し、各パターンの扱いを明示する。

## STEP3 デグレを起こさないことを検証する

デグレを起こさないことの確認は**推論だけに頼らない**。

- 既存テストの実行、再現テストおよび周辺パターンのテスト追加で裏付ける。
- テストがない／書けない箇所は**「未検証」であることを明示する**。

### 検証
- 証跡：STEP1 の再現テストが**赤→緑**に変わり、かつ**既存スイートが緑**のままであること。
- 実行は普遍化できないためプロジェクトのテストコマンドに委譲する。アプリ実行確認は `/verify`／`/run`、
  差分のバグ観点レビューは `/code-review` を併用してよい。
- 書けない箇所は「未検証」と明示し、残存リスクとして STEP5 に集約する。

## STEP4 根本対策で変更を最小にする

- 対策は根本原因に対する**根本対策**とする。症状だけを抑える場当たり的な対症療法（ローカルなパッチ）は、
  設計をツギハギにして技術的負債やデグレにつながるため選ばない。
- そのうえで、根本的・全体的な設計まで含めて検討し、根本対策を満たす範囲で変更を最小にする。
  根本対策に設計レベルの変更が必要なら、それを含めた範囲が最小単位であり、削るのは過小対応（対応漏れ）。
  逆に、原因と無関係な変更・先回りの一般化・ついでのリファクタなど根本対策に不要な上乗せはしない
  （＝「過剰な対応をしない」の意味）。

### 検証
- 証跡：`git diff --stat` で**変更ファイル/hunk が原因スコープに収まる**ことを確認。原因と無関係な
  変更・ついでのリファクタ（過剰）も、根本対策に必要なのに削った箇所（過小）も無いこと。

## STEP5 確認できない範囲はユーザーに判断を仰ぐ

- 全影響の把握やデグレ無しの確認が難しい場合は、確認できた範囲とできなかった範囲（残存リスク）を明示し、
  ユーザーに判断を仰ぐ。

### 検証
- 証跡：「**確認済み／未確認(残存リスク)**」を構造化して提示する。各 STEP で「未検証」とした項目を
  ここに集約し、ユーザーが判断できる状態にする。
