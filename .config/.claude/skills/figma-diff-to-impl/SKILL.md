# Figma差分 → 実装計画スキル

FigmaのBefore/Afterデザインを比較し、リポジトリの技術スタックに合わせた実装計画を作成する。

## アーキテクチャ方針

**全フェーズをサブエージェントに委譲する。**
各フェーズの詳細な作業（大量のAPI呼び出しとそのレスポンス）はサブエージェントのコンテキスト内で完結させ、メインセッションには**圧縮された成果物だけ**を返す。これによりメインセッションのコンテキスト消費を最小限に抑える。

```
メインセッション（成果物の受け取りと次フェーズへの引き継ぎのみ）
 ├─ フェーズ0エージェント → 技術スタック判定結果（数行）
 ├─ フェーズ1エージェント → ノードID一覧・対応関係の表
 ├─ フェーズ2エージェント → 差分一覧表
 └─ フェーズ3エージェント → 実装計画ドキュメント
```

### サブエージェントの起動方法

各フェーズは Claude Code の **`Task` ツール**で起動する。
専用の定義ファイルは不要で、プロンプト文字列を渡すだけでよい。
Figma MCP を含む利用可能なツールはサブエージェントに自動的に引き継がれる。

```
Task(prompt="（各フェーズの「エージェントへの指示」をここに展開する）")
```

フェーズは必ず**順次実行**すること（前フェーズの返り値を次フェーズに渡すため、並列実行は不可）。

---

## メインセッションの進行手順

以下の順で `Task` ツールを使いサブエージェントを順次起動する。
各エージェントには「エージェントへの指示」セクションの内容をプロンプトとして渡すこと。
前フェーズの返り値は次フェーズのエージェントへの指示に含めて引き継ぐこと。

### フェーズ 1：構造把握エージェント

**起動方法**

**前提**: フェーズ0の返り値をプロンプトに含めて渡すこと。

**返り値フォーマット**


**エージェントへの指示**

```

Before/Afterの対応関係を整理して返してください。
説明・感想・補足は一切不要です。

Figma URL: {{FIGMA_URL}}

## 手順

### Step 1: 全体構造の把握
FigmaURLに含まれる node-id パラメータ（例: ?node-id=780-13809）を抽出し、
- を : に変換したもの（例: 780:13809）を nodeId として get_metadata を実行する。
※ get_metadata を nodeId なしで呼び出してはいけない。

取得結果から全セクション・全フレームのノードID一覧を整理する。

### Step 2: スクリーンショット取得
Step 1 で特定した全セクションに対して get_screenshot を実行する。
Before / After の対応関係（どの画面がどの画面に対応するか）を整理する。

## 厳守ルール
- ツール呼び出しをスキップしてはいけない
- 全ノードに対して必ずツールを実行し、実測値をもとに判断すること

## 返り値フォーマット（このフォーマット以外を返してはいけない）

【ノードID一覧】
| セクション名 | フレーム名 | ノードID | Before/After |
|-------------|-----------|---------|--------------|

【Before/After 対応関係】
| Before 画面名 | After 画面名 | ノードID(Before) | ノードID(After) |
|--------------|-------------|----------------|----------------|
```

```
## 手順

全フレームのペアに対して、BeforeとAfterそれぞれに get_design_context を実行し、
以下の項目を比較・記録する。

| 比較項目 | 確認ポイント |
|----------|-------------|
| テキスト内容 | 追加・削除・変更 |
| テキストカラー | カラーコードで比較（例: #888888 → #282828） |
| フォントサイズ・ウェイト | 数値で記録 |
| 余白・レイアウト | padding / margin / gap の変化 |
| 要素の表示条件 | 条件付き表示の変化 |
| data-development-annotations | デザイナーの実装向け注記。必ず確認する |

## 厳守ルール
- ツール呼び出しをスキップしてはいけない
- 1画面でも省略した場合、差分の洗い出しは完了とみなさない
- 変更なしの画面も「変更なし」として表に明記すること
```

```
## 手順

### Step 5: 実装ファイルへのマッピング
差分一覧の各変更箇所について、対応するファイル・クラス・行番号を特定する。

1. 画面名・テキスト・コンポーネント名でリポジトリを検索し、対象ファイルを特定する
   例: grep -rn "対象テキストやクラス名" src/

2. 文字列リソースの変更が必要な場合は該当キーとファイルを特定する

3. 既存のユーティリティ・共通コンポーネントが再利用できるか確認する

4. 変更が複数ファイルに連鎖する場合（共通コンポーネントなど）も明記する

技術スタック別の確認観点（技術スタック判定結果に合わせて適用する）:
- Android / Compose : @Composable 関数名、strings.xml のキー、Color.kt の色定数
- iOS / SwiftUI     : View 構造体名、Localizable.strings のキー、Assets.xcassets のカラーセット
- React / Next.js   : コンポーネントファイル名、CSS Modules / Tailwind クラス、i18n キー
- Flutter           : Widget クラス名、l10n/*.arb のキー、ThemeData の色定義
- その他            : 技術スタック判定結果から適切な観点を自律的に導くこと
```
