# コードレビュースキル

このスキルが呼ばれたら、以下のレビュー観点とスタイルに従ってコードレビューを実施してください。
引数としてPR番号やファイルが渡された場合はその対象をレビューし、なければ現在のブランチの差分をレビューしてください。

---

## レビュースタイルの原則

1. **質問形式で指摘する**
   - 断定的に「これは間違い」とは言わず、「〜はいかがでしょうか？」「〜という背景はございますでしょうか？」と問いかける
   - 自分の理解が正しいか確認する質問と、改善提案の質問を明確に区別する

2. **背景・理由を必ず添える**
   - 「なぜそう思うか」を丁寧に説明する
   - 例：「仮にこのダイアログに改修が発生した場合、2か所にあることで修正漏れが起きそうな懸念があります」

3. **実装案・代替案を提示する**
   - 問題を指摘するだけでなく、コードスニペットで代替実装を示す
   - 「こんな風に書けないでしょうか？」という形で提案する

4. **修正の強制度を明示する**
   - 必須修正：「〜をお願いします」
   - 任意修正：「このままでも問題ありません」「軽微なのでスルーでも問題ありませんが」
   - 確認だけ：「〜という背景はどのような場合でしょうか？」

5. **細かい指摘には謙遜を添える**
   - 「めっちゃ細かいのですが」「細かくて恐縮ですが」など、些細な指摘であることを先に断る
   - 自分の理解が浅い可能性を認め、「私の理解が浅くてご質問です」と添える

6. **他メンバーの意見を求める**
   - 判断が難しい場合は「どう思いますか？」と問いかける

7. **誤解・ミスを素直に認める**
   - 自分の指摘が間違いだった場合は素直に認め、謝罪する

## レビュー観点チェックリスト

### 1. 命名規則

- **typoをチェックする**
  - スペルミスを「typoかもしれません」と確認形式で指摘する
  - コメント文中のtypoも対象にする

- **命名の具体性・適切な抽象度**
  - 広すぎる（汎用すぎる）場合は具体化を促す
  - 特定の用途に限定されすぎる場合は汎用化を提案する

- **プロジェクト内の命名の一貫性**
  - 既存コードの命名規則に合わせることを求める
  - 同じ概念に複数の命名が混在している場合は統一を求める（例：`EKYC, Ekyc, EKyc` の混在）

- **実態に合った命名**
  - メソッド名と処理内容が一致しているか確認する
  - スコープを明確にする命名（例：`QuickDepositBank` のようにコンテキストを含める）

- **その他確認ポイント**
  - `@StringRes` などのアノテーションが付いているか
  - `Author` 情報など不要なコメントが残っていないか
  - 廃止されたサービス名（Fabric等）が残っていないか

---

### 2. コードの可読性

- **ネストを減らす・早期リターンを使う**
  - 例：`val userId = userIdUseCase.get().first() ?: return`

- **肯定を先に処理する**
  - `isAllValid` のようなメソッド名なら内部ロジックも肯定を軸に統一する
  - 例：`viewType.value = if (hasAutoDepositCourse) { … }`

- **if-else を `let` などで簡潔にする**
  - `null` チェックを `?.let { … }` で書き換えるとシンプルになる場合を指摘する

- **冗長なコードを整理する**
  - ほぼ同じ処理が2か所に定義されている場合は統合を提案する
  - `private val` が init でしか使われていない場合は引数への変更を提案する
  - `val` になった場合に `get()` を省略できるか確認する

- **trailing comma の追加**
  - Kotlin のコーディング規約に従い、末尾カンマの追加を促す

---

### 3. 設計・アーキテクチャ

- **共通化の判断基準**
  - 再利用が見込めるか、共通化によって設計思想が分かりにくくならないか
  - 「事前に予測した自由度で設計を複雑にするより、要件が出た時に対応する方が良い」（YAGNI原則）

- **Widgetのパディング設計**
  - Widget の外側にパディングを持たせる場合は、上下左右全て統一する
  - 「上/右/左にはPaddingがあるけど下だけない」ような中途半端な設計は指摘する
  - 推奨：パーツ側はパディングを持たず、利用側で付けてもらう設計

- **責務の分離**
  - 画面ロジック（条件分岐）が View/Composable に含まれている場合は ViewModel 側に移動を促す
  - モデルにビジネスロジックを持たせず、UseCase に実装することを推奨する

- **デッドコードの除去**
  - どこからも呼び出しがないコードは積極的に削除するか指摘する
  - 未使用の import、フィールド、メソッドを確認する

---

### 4. Androidアーキテクチャ固有

#### LaunchedEffect / UiEvent
- `screenState` の中にある値を `LaunchedEffect` で処理している場合、`Composable` 関数で処理すべきでないか確認する
- `LaunchedEffect` のキーは適切か確認する（`Unit` にして良いケース vs 可変値が必要なケース）

#### ViewModel / UseCase
- 画面ログ（Screen Log）は ViewModel の `init` で送信する設計方針に準拠しているか
- `UseCase` 命名に HTTP メソッド名（`Get`, `Post`）が含まれていないか（ドメイン観点の命名が適切）
- バリデーション結果（Valid/Invalid）は ViewModel または UseCase 側に持たせ、画面側に持たせない

#### StateFlow / Flow
- `StateFlow` の初期値は適切か（`null` が不要な場合は `Unit` や適切なデフォルト値にする）
- イベント消費（consume）の順序：処理を実行してから消費する

#### Jetpack Compose
- Preview は適切なコンテンツが揃っているか（他と一貫しているか）
- `@Preview(showBackground = true)` の付与を確認する
- Fake 実装クラスは Preview メソッド内に定義する方がわかりやすい
- `Modifier.fillMaxWidth()` でタップ領域を画面端まで広げる（テキストリンク等）
- `Visibility.invisible` の代わりに `alpha = 0f` を使う

#### Fragment / Activity
- `Intent` の `addFlags` など意図的な設定が消されないよう、コメントを残す
- `private` にできるフィールドは `private` にする

---

### 5. コードの品質

- **null 安全の扱い**
  - `null` を引数に渡すことで実質「何もしない」処理になっていないか確認する
  - Kotlin の `?.let` や `?: return` で null チェックを簡潔に書く

- **未使用コード・意図せず含まれた差分**
  - 変更していないはずのファイルが差分に含まれていたら確認する

- **安全性の考慮**
  - メインスレッドをブロックする処理が ANR リスクにならないか確認する
  - Crashlytics へのエラー送信が漏れていないか確認する

- **文字列リソースの扱い**
  - 既存の文字列定義（`strings.xml`）と同じ文言を直書きしている場合は既存定義の使用を促す

- **バリデーション**
  - 条件の `<` と `<=` の正確性を確認する
  - 状態の意味とロジックの一致を確認する（`NotSet` が `Valid` になるか `Invalid` になるかなど）

---

### 6. コメント・ドキュメント

- **WHY を書く（理由・背景の記述）**
  - 「何をしたか」の事実ではなく「なぜそうしたか」の根拠を書くことを強く求める
  - コードを読めばわかる結果ではなく、コードからは読み取れない背景を書くよう促す

- **実態との乖離を防ぐ**
  - コメントの内容がコードの実態と合っているか確認する
  - メソッドを大きく書き換えた場合はコメントやメソッド名も更新を促す

- **未来の開発者への配慮**
  - 特殊な処理は、消してしまうと問題が起きることを説明するコメントを残す

- **TODOコメント**
  - 今回対応できない部分は TODO コメントとして残すことを提案する

- **AIへの指示の明確化（Copilot / Claude 対象）**
  - 曖昧さがある場合は具体化を求める（「適切というのが曖昧さを含んでいる」）
  - 原則に例外がある場合は例外を明記する（「AIは例外を理解できないため」）

---

### 7. テスト・品質担保

- テストケースが重複していないか確認する
- テストで動作確認できることとできないことを区別する
- リリースビルドでの動作確認が必要なケースは事前確認を促す

---

## レビュー時の優先度判断

| 優先度 | 内容 | 対応例 |
|--------|------|--------|
| 高（必須修正） | バグ・仕様漏れ・クラッシュリスク | 「〜お願いします」 |
| 中（推奨修正） | 可読性・命名・設計一貫性 | 「いかがでしょうか？」 |
| 低（任意） | スタイル・細かいリファクタリング | 「このままでも問題ありません」 |
| 確認のみ | 実装の意図・背景の理解 | 「〜という背景をお伺いできますでしょうか？」 |

---

## 出力形式

レビュー結果は以下の形式でまとめてください：

1. **必須修正** - バグ・仕様漏れなど絶対に直すべき点
2. **推奨修正** - 品質向上のための改善提案
3. **任意修正** - あると良いが強制しない細かい点
4. **確認事項** - 意図を確認したい点・質問

各指摘は「対象箇所 → 観点 → 提案/質問」の形で記述し、必要に応じてコードスニペットを添える。
