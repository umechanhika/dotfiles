# コードレビュースキル

このスキルが呼ばれたら、以下の実行フローに従ってコードレビューを実施してください。
引数としてPR番号やファイルが渡された場合はその対象をレビューし、なければ現在のブランチの差分をレビューしてください。

---

## 実行フロー

### STEP 1: 現在のブランチのPR確認

```bash
gh pr view --json number,title,url,headRefName,author
```

- PRが存在しない場合 → STEP 3へ（差分のみレビュー）
- PRが存在する場合 → STEP 2へ

### STEP 2: PRの作成者確認（自分のPRかどうか）

```bash
gh api user --jq '.login'
```

取得した `login` と STEP 1 の `author.login` を比較する。

- **自分が作成したPR**: STEP 3・4・5a のみ（未解決コメントへの対応方針のみ。新規指摘は行わない）
- **他者が作成したPR**: STEP 3・5b のみ（新規指摘のみ。STEP 4・5a はスキップ）
- **PRなし**: STEP 3・5b のみ（新規指摘のみ）

### STEP 3: コード差分の取得

```bash
# PRがある場合
gh pr diff {pr_number}

# PRがない場合
git diff origin/main...HEAD
```

### STEP 3.5: パターン検索（構文で識別できる観点の事前確認）

差分に含まれる Kotlin ファイルを対象に、LLM の注意が分散しやすいパターンを grep で機械的に抽出する。
STEP 5b の新規指摘生成時に、この検索結果を必ずインプットとして参照すること。

```bash
# 差分に含まれる Kotlin ファイルを取得
gh pr diff {pr_number} --name-only | grep "\.kt$"

# 各ファイルに対して以下を実行
grep -n "@SuppressLint\|@Suppress(" {file}
grep -n "remember {" {file}
grep -n "DialogProperties" {file}
```

各ヒットに対する判断基準:

| パターン | 確認内容 | 指摘候補になる条件 |
|---------|---------|-----------------|
| `@SuppressLint` / `@Suppress(` | 直前行に `//` コメントがあるか | コメントがない場合 → コードの品質「アノテーションのコメント」として指摘 |
| `remember {` | キーなしで外部の引数（ViewModel・コールバック等）を内部に保持しているか | 保持している場合 → Jetpack Compose「rememberキー指定」として指摘 |
| `DialogProperties` | `dismissOnBackPress` / `dismissOnClickOutside` が明示されているか | 明示なし（デフォルト）かつ旧実装（`isCancelable`）から動作が変わる場合 → 確認事項として指摘 |

### STEP 4: 未解決コメントの取得（自分が作成したPRの場合のみ）

GitHub GraphQL API で未解決のレビュースレッドを取得する：

```bash
gh api graphql -f query='
{
  repository(owner: "{owner}", name: "{repo}") {
    pullRequest(number: {pr_number}) {
      reviewThreads(first: 100) {
        nodes {
          isResolved
          id
          comments(first: 10) {
            nodes {
              id
              databaseId
              body
              author { login }
              path
              line
              originalLine
              url
            }
          }
        }
      }
    }
  }
}'
```

`isResolved: false` のスレッドのみ対象とする。
owner と repo は `gh repo view --json owner,name` で取得する。

### STEP 5: レビュー実施

**a. 未解決コメントへの対応検討**（自分が作成したPRの場合のみ）
- 各コメントの内容を読み、差分コードと照らし合わせて対応要否を判断する
- 対応方針と返信案を作成する（後述のスタイル原則に従う）

**b. 差分コードへの新規指摘生成**（他者のPR / PRなしの場合のみ）

単一パスでは観点の取りこぼしが起きやすいため、観点を3つに分担した**サブエージェントを3つ並行起動**して洗い出し、その結果を親（あなた）が統合する。

#### b-1. 3つのサブエージェントを並行起動する

`Agent`（`subagent_type` は `general-purpose`）を**1メッセージで3つ同時に**起動する。各サブエージェントには、担当観点・重点的に読む参照ファイル・差分・STEP 3.5 のパターン検索結果・指摘記述ルール（b-3）を渡す。

| エージェント | 担当観点 | 重点的に読む参照ファイル |
|------------|---------|----------------------|
| Agent A: 設計・構造系 | 設計/アーキテクチャ、コルーチン・Flow、パフォーマンス、責務分離、共通化(YAGNI)、デッドコード | `references/review-general.md`（3・8・9・11）／`references/review-android.md`（2・4） |
| Agent B: UI・可読性系 | Jetpack Compose、命名、可読性、Kotlin イディオム、アクセシビリティ、テーマ、プレビュー | `references/review-general.md`（1・2・10・15）／`references/review-android.md`（1・3・11・12・13） |
| Agent C: 品質・安全系 | コード品質、null安全、コメント/ドキュメント、セキュリティ、テスト、ライフサイクル/リーク、R8、Room | `references/review-general.md`（4・5・6・7・12・13・14）／`references/review-android.md`（5・6・7・8・9・10・14） |

各サブエージェントへのプロンプトに必ず含める:
- レビュー対象の差分（`gh pr diff {pr_number}` の結果。サブエージェント側で再取得させてもよい）
- 担当観点と、重点的に読む参照ファイルの**絶対パス**（このスキルの `references/` 配下。例: `/Users/hikaru.umetsu/.claude/skills/pre-code-review/references/review-general.md`）
- STEP 3.5 のパターン検索結果
- 「各指摘に ファイル・行を付ける。行は `grep -n` で実ファイル行を確認する」指示
- b-3 の指摘記述ルール（既存コードを引用しない／suggestion は変更行のみ／言い回しルール／**重要度ラベルは書かない**）
- 「担当観点で該当がなければ『指摘なし』と返す」指示

#### b-2. 親が統合する

3エージェントの結果を受け取ったら:
- **同一箇所・同一趣旨の重複指摘を1件に集約する**（最も的確な記述を採用）
- 各指摘のファイル・行を必要なら `grep -n` で再確認する
- 各指摘を b-3 の言い回しルールで最終整形する
- 修正を強く促すもの（バグ・クラッシュ・仕様漏れ）を先に、確認・任意の指摘を後に並べる

#### b-3. 各指摘の記述ルール（親・サブエージェント共通）

指摘ごとに、内容を書く前に**トーン**を決める。トーンは指摘の性質で決まり、**出力にはラベルとして書かず**、言い回しだけで伝える:
1. 指摘の性質を見極める（バグ・必須の修正依頼／設計・可読性の提案／意図の確認／任意の微修正）
2. 後述の表でそのトーンの「使う表現」を確認し、**締め文のパターン**を頭に入れる
3. 指摘の内容を書き、確認したパターンで締める
4. 書き終えたら**締め文が表と一致しているか確認する**（提案のつもりで「できないでしょうか？」が出たら書き直す）

**既存コードの引用について:**
`suggestion` ブロックは提案コードのみに使い、**既存コードはコードブロックで引用しない**。
「現在の実装は以下の通りです」と書いて既存コードを貼りたくなった場合は、コードを貼らず「〜という実装になっていますが」と言葉だけで状況を説明する。
レビュアーはすでにコードを手元で見ているため、既存コードの引用はノイズになる。

**suggestion には変更行のみを含める:**
GitHub の `suggestion` ブロックは、**コメントを紐付けた行範囲をその内容で丸ごと置換**する。
そのため「変更したい行」「コメントの紐付け範囲」「suggestion の行数」の3つを一致させる必要がある。
前後の未変更行まで suggestion に含めると、紐付け範囲とずれてコードが重複・破損し、レビュアーも差分の意図を読み取りにくくなる。

- **1行で済む変更** → その1行にコメントを紐付け、suggestion にはその行の修正版だけを入れる。
- **連続する複数行の変更** → その範囲（`start_line`〜`line`）にコメントを紐付け、suggestion にはその範囲分だけを入れる。

例えば次のような複数引数のコンストラクタで、2番目の引数だけ修正すればよい場合:

```kotlin
val point = Point(
    x = origin.left,
    y = origin.left,  // ← origin.top の誤りで、ここだけ直したい
    z = origin.depth,
)
```

- NG: コンストラクタ全体（`Point(` から `)` まで）を suggestion に入れる。
- OK: `y = origin.left,` の1行にコメントを紐付け、suggestion はその行の修正版だけにする。

```suggestion
    y = origin.top,
```

- レビュー観点は `references/review-general.md`（言語非依存）と `references/review-android.md`（Android/Kotlin/Compose 固有）に集約してある。b-1 の担当割当に従い各サブエージェントが参照する
- 各指摘には **ファイル** と **行** を必ず記載する（行が特定できない場合は `-`）
  - `gh pr diff` の出力はdiff上の行番号であり、実際のファイルの行番号と異なる。**実際のファイル行番号**を取得するには `grep -n "{特徴的なコード}" {ファイルパス}` で確認する
  - 行番号を誤って記載するとレビュアーが該当箇所を探せないため、確認できない場合は `-` にする
- **指摘事項は、修正を強く促すもの（バグ・クラッシュ・仕様漏れ）から先に、確認・任意の指摘を後に並べる**（重要度ラベルは出力しない）
- 指摘のトーンごとの言い回しは以下のルールに従う。**この表の「必須修正/推奨修正/任意修正/確認のみ」は出力に書くラベルではなく、言い回しを選ぶための内部分類である。**

| トーン（内部分類） | 使う表現（具体例） | 禁止表現（絶対使わない・具体例） |
|--------|----------|--------------------------|
| 必須修正 | 「〜に変更できないでしょうか？」「〜に修正できないでしょうか？」（**締め文は必ずこの形で終える**。「〜はいかがでしょうか？」で終えたくなった場合はトーンを推奨修正に変更すること） | 「〜していただけますでしょうか？」「ご確認いただけますでしょうか？」「確認させてください」「ご確認のうえ〜」「〜はいかがでしょうか？」（これは推奨修正の表現） |
| 推奨修正 | 「〜はいかがでしょうか？」「〜という形はどうでしょう？」（**締め文は必ずこの形で終える**。確認事項の表現で終えたくなった場合はトーンを確認事項に変更すること） | 「〜できないでしょうか？」（**動詞の種類・形を問わず「〜できないでしょうか？」はすべて禁止**。「変更できないでしょうか？」「統一できないでしょうか？」「抽出できないでしょうか？」「分けられないでしょうか？」「まとめられないでしょうか？」「移動できないでしょうか？」など形が変わっても同様に禁止。推奨修正は「提案についてどう思うか」を問うので「いかがでしょうか？」を使う）「〜していただけますでしょうか？」「意図的な設計でしょうか？」「意図はございますでしょうか？」（これらは確認事項の表現） |
| 任意修正 | 「スルーでも問題ありませんが〜」「このままでも問題ありませんが〜」で**必ず書き始める** | 「〜修正していただけますでしょうか？」「〜できないでしょうか？」 |
| 確認のみ | 「意図的な設計でしょうか？」「〜という意図はございますでしょうか？」「〜という理解でよろしいでしょうか？」（**文末は必ず「〜でしょうか？」の疑問文で終える**） | 「〜できないでしょうか？」「確認できないでしょうか？」（「〜できないでしょうか？」の形はすべて禁止）「〜していただけますでしょうか？」「確認させてください」「〜確認しておきたいのですが」「〜ください」「〜お願いします」（確認事項は相手に行動を求めるのではなく、意図・理解を問う純粋な疑問文にする） |

> **重要**: この言い回しルールは指摘の締め文だけでなく、**指摘内容の本文（改善案・代替案の提示部分）にも適用する**。
> - 確認事項の本文内に改善案を添える場合は「〜という方法も考えられます」「〜という形も一案ですが」にとどめ、「〜できないでしょうか？」は使わない。
> - 「確認できないでしょうか？」は「〜できないでしょうか？」の一形態であり、確認事項・推奨修正・任意修正での使用は**禁止**。必須修正でのみ使用可能だが、その場合も「確認」ではなく「修正・変更」を求める動詞を使う。

**各トーンの概念的な違い（表現選択に迷ったときの判断基準）:**
- **必須修正** → 「この問題を修正することは可能ですか？（可否を問う）」→「〜できないでしょうか？」
- **推奨修正** → 「この提案についてどう思いますか？（意見を問う）」→「〜はいかがでしょうか？」
- **任意修正** → 「問題はないが、もし良ければ（提案を先に否定しない）」→「このままでも問題ありませんが〜」で始める
- **確認事項** → 「これは意図的ですか？（意図・理解を問う）」→「〜でしょうか？」の疑問文

推奨修正のつもりで「〜できないでしょうか？」が浮かんだら、それは必須修正か確認事項の表現なのでトーンの見直しを検討すること。

### STEP 5.5: 出力前セルフチェック

mdファイルを書く前に、生成した各指摘を次の観点で確認する:

- [ ] 推奨修正の指摘が「〜はいかがでしょうか？」「〜どうでしょう？」で終わっている（「できないでしょうか？」が入っていたら書き直す）
- [ ] 既存コードをコードブロックに引用していない（`suggestion` ブロックは提案コードのみ）
- [ ] `suggestion` に変更行のみを含めている（前後の未変更行を含めていない）。suggestion の行数がコメントの紐付け行範囲と一致している
- [ ] 確認事項の指摘が「〜でしょうか？」の疑問文で終わっている（「確認させてください」「〜ください」は書き直す）
- [ ] 任意修正の指摘が「このままでも問題ありませんが」または「スルーでも問題ありませんが」で始まっている
- [ ] 必須修正・推奨修正・確認事項の指摘内容の最初の文が「？」で終わっている（「〜です。」「〜ます。」「〜います。」で始まっていたら書き直す）

### STEP 6: mdファイルの生成・配置

```bash
mkdir -p {カレントディレクトリ}/tmp/review
```

ファイル名: `review_{pr_number}_{title}.md`（PRなしの場合は `review_branch_{YYYYMMDD_HHMMSS}.md`）

`{title}` は `gh pr view --json title --jq '.title'` で取得したPRタイトル文字列をそのまま使う。
ファイル名に使えない文字（`/`・macOS で問題になる `:` など）は `_` に置換する。

後述の出力フォーマットに従いmdファイルを作成し配置する。

### STEP 7: 結果の報告

- 作成したmdファイルのパスをユーザーに伝える
- 未解決コメントがあった場合は件数と対応要否の内訳を簡潔に伝える
- 「コメントを投稿する場合は指示してください」と添える（投稿時は後述の必須ゲートで、最新 md を再読込し全文を表でプレビューしてから承認を得て投稿する）
- PR がない場合（Mode C）は投稿先がないため、md 出力までで完了する

## レビュースタイルの原則

このスタイルは **新規指摘コメント** と **未解決コメントへの返信案** の両方に適用する。

1. **優先度に応じた言い回しを使い分ける**
   - 実行を求める場合は「〜していただけますでしょうか？」（実行有無を問う）ではなく「〜できないでしょうか？」（実行可否を問う）を使う
   - 相手が何らかの理由でできない可能性がある場合、実行有無を問うと主張が衝突するリスクがある
   - 推奨・任意・確認の指摘: 断定せず「〜はいかがでしょうか？」「〜という背景はございますでしょうか？」と問いかける
   - 自分の理解が正しいか確認する質問と、改善提案を明確に区別する

2. **「提案→理由」の順で書く**
   - 構成の基本形: まず提案・質問を書き、その後に理由・背景を1文で添える
   - **現状の実装を説明する文（「〜という実装になっています」「〜が〜されています」「〜状態です」）は最初に書かない**。コードを見れば分かる情報であり、理由の説明で補完できる。

   **推奨修正の例:**
   - NG: 「`Content.Launched` と `Content.OnLoad` で同じ引数列が2か所に重複しています。フィールドを追加したときに片方が漏れやすいので、共通関数に切り出す形はいかがでしょうか？」
   - OK: 「共通関数に切り出す形はいかがでしょうか？`Content.Launched` と `Content.OnLoad` で同じ引数列が2か所に重複しており、フィールドを追加したときに片方が漏れやすいためです。」

   **確認事項の例（現状説明を疑問文に折り込む）:**
   - NG: 「`dismissOnBackPress` がデフォルト（true）になっているため、バックキーでダイアログを閉じられる状態です。旧実装の `isCancelable = false` から意図的に変更したのでしょうか？」
   - OK: 「`isCancelable = false` を外して `dismissOnBackPress` をデフォルト（true）にしたのは意図的でしょうか？」（「何を変えたか」を疑問文の主語にする）
   - NG: 「`remember` にキーが渡されていないため、引数が変わっても再生成されません。意図的な設計でしょうか？」
   - OK: 「`remember` にキーを渡さない設計は意図的でしょうか？引数が変わっても再生成されないためです。」

   - 理由がコードスニペットや質問文で伝わる場合は省略してよい

3. **実装案・代替案を提示する**
   - コードスニペット（suggestionブロック）は提案コードのみに使う。既存コードはソースコードを見れば分かるため引用しない
   - typoなど受け取り手が自己解決できる指摘は、修正案を書かずに指摘のみにする
   - 提案は最小限の修正を優先する。大規模な設計変更は「〜という方法も考えられますがいかがでしょうか？」と軽く触れる程度にとどめる

4. **修正の強制度を明示する**
   - 必須修正：「〜できないでしょうか？」
   - 任意修正：「このままでも問題ありません」「軽微なのでスルーでも問題ありませんが」
   - 確認だけ：「〜という背景はどのような場合でしょうか？」

5. **細かい指摘には謙遜を添える**
   - 「めっちゃ細かいのですが」「細かくて恐縮ですが」など、些細な指摘であることを先に断る
   - 自分の理解が浅い可能性を認め、「私の理解が浅くてご質問です」と添える

6. **相手に確認を促す**
   - 判断が難しい場合は「〜と思うのですがいかがでしょうか？」「〜はどうでしょうか？」と問いかける
   - 返信案でも自分の方針・意図・背景を示したうえで、「〜という認識でよろしいでしょうか？」「いかがでしょうか？」と添える

7. **誤解・ミスを素直に認める**
   - 自分の指摘が間違いだった場合は素直に認め、謝罪する

8. **文章は簡潔に書く**
   - 説明口調・冗長な言い回しを避ける
   - 背景や理由は添えるが、一文で収まる程度に簡潔にする
   - 「1点目・2点目」のような番号付き構造化はしない。1指摘につき観点は1つに絞り、複数の懸念がある場合はより重大なものだけを指摘する

9. **「〜べき」という表現は使わない**
   - 背景や意図を確認していない状態で断言できないため
   - 代わりに「〜はいかがでしょうか？」「〜という形にするのはどうでしょう」と提案形にする

10. **複数の文を続ける場合は文ごとに改行する**
    - 一文ごとに改行し、詰め込みすぎない

11. **断定せず、可能性の提示をする**
    - 「〜かもしれません」「〜ような気がします」「〜と思われます」などの表現を使う
    - 自分の判断が誤っている可能性を常に考慮する

12. **一度記載した情報は繰り返さない**
    - 指摘の説明内で同じ内容を再掲しない

## 未解決コメントへの対応方針の判断基準

| 判断 | 内容 |
|------|------|
| **対応必要** | コメントの指摘が妥当で、差分コードに修正の余地がある |
| **対応不要** | すでに修正済み、仕様上意図的な実装、またはコメントの前提が誤っている |
| **要確認** | コメントの意図が不明確、または判断が難しい |

下表は**指摘を並べる順序と言い回しを決めるための内部判断軸**であり、出力にラベルとして書かない。

## PRへのコメント投稿

ユーザーから「投稿して」「コメントして」「返信して」などの指示があった場合のみ実施する（PR がない Mode C は投稿先がないため対象外）。

投稿は必ず下記「投稿前の確認と承認（必須ゲート）」を通してから行う。**承認を得るまで投稿 API は実行しない。** これがこの投稿フローの中核となる不変条件である。

### 投稿前の確認と承認（必須ゲート）

「投稿して」の指示を受けても、いきなり API を呼ばない。次の手順で投稿予定の内容を提示し、`AskUserQuestion` で承認を得てから投稿する。`pre-commit-leak-check` の承認ゲートと同じ考え方で、版ズレ（ユーザー編集の取りこぼし）・誤投稿・重複投稿を投稿前に止めるための関門である。

#### 手順

1. **md をディスクから読み直す（必須・直前。`cat` を使う）**
   コンテキストに残っている版を信用せず、対象の md ファイル（STEP 6 で出力したもの）の**ディスク実体を `cat` で読み直す**。以降の表・本文・API ペイロードは、この `cat` で得た内容からのみ組み立てる。パスに `[ ] ( )` などシェルのグロブ文字が含まれることがあるため、必ずダブルクォートで囲む。

   ```bash
   cat -n "{mdの絶対パス}"
   ```

   **なぜ Read ではなく `cat` を使うのか（このゲートの肝）:**
   Read ツールは「同一セッションで既に読んだファイルは変わっていない」と判断すると `Wasted call — file unchanged since your last Read` を返し、**ディスクの最新バイトではなくコンテキスト上の古い版に居座らせる**ことがある。md は生成後にユーザーが IDE で編集（指摘の削除・文面修正）するのが想定運用で、その編集を Read が取りこぼすと、まさにこのゲートが防ごうとしている版ズレ（削除したはずの指摘を投稿してしまう等）がそのまま起きる。`cat` はキャッシュ判定を介さずディスクのバイトをそのまま返すため、外部編集を確実に拾える。**ここは「専用ツール（Read）では目的を達成できない」ことが分かっている場面なので、`cat` の使用が正当化される**（通常 `cat` より専用ツールを優先する原則の例外）。

   - レビュー生成後にユーザーが編集している前提で読む。特に system-reminder に「opened/編集中/modified」が出ている場合は省略せず `cat` で実体を確認する。
   - `cat` の内容がコンテキスト上の記憶と食い違ったら、**必ず `cat` の内容を正**として扱い、表・全文プレビュー・ペイロードを作り直す。記憶側を優先しない。

2. **本文は md から逐語で組み立てる**
   投稿する本文は再 Read した md のテキストを**そのまま使う（要約・言い換え・再生成をしない）**。以前に書いた JSON などの中間生成物は再利用しない（古い版の混入を防ぐ）。必要なら現在の md から作り直す。

3. **投稿方式を判定する**
   最新のコミットSHAと差分ハンクを取得し、各指摘の行が差分内かを確認する。

   ```bash
   gh pr view {pr_number} --json headRefOid --jq '.headRefOid'
   gh pr diff {pr_number} | grep "^@@"
   ```

   GitHub REST API の行コメントは差分ハンクに含まれる行（変更行＋前後のコンテキスト行）にしか投稿できない。
   - 行が差分内 → `インライン`
   - 行が差分外、または行を特定できない → `PRコメント(行特定不可)`（承認時に PR 全体コメントとして投稿する）

   `suggestion` を含む指摘は、suggestion の行数とコメント紐付け行範囲を一致させる（1行の修正は `line` のみ、連続複数行は `start_line`〜`line`）。一致していなければこの時点で直す。

4. **表を提示する**（投稿対象の索引。再 Read した md から生成する）

   新規指摘（他者PR・Mode B）の場合:

   ```
   ## 投稿内容の確認（新規指摘 N 件）

   | # | ファイル | 行 | 投稿方式 | 指摘内容（冒頭） |
   |---|---------|----|---------|----------------|
   | 1 | path/to/A.kt | 42 | インライン | 共通関数に切り出す形はいかがでしょうか？… |
   | 2 | path/to/B.kt | - | PRコメント(行特定不可) | … |
   ```

   未解決コメントへの返信（自PR・Mode A）の場合:

   ```
   ## 投稿内容の確認（返信 N 件）

   | # | ファイル:行 | 元コメント(@author) | 対応要否 | 返信内容（冒頭） |
   |---|------------|--------------------|---------|----------------|
   | 1 | A.kt:42 | @reviewer: … | 対応必要 | …という対応をしようと思いますが… |
   ```

   - 行は単一行または `start-line`、特定できない場合は `-`。投稿方式は `インライン` か `PRコメント(行特定不可)` のいずれか。
   - 「冒頭」列は一覧用の要約（1文程度、長ければ `…`）。**正確な内容は必ず次の全文プレビューで確認する**（要約だけで承認させない）。

5. **投稿プレビュー（全文）を提示する**
   表の直後に、各指摘について**実際に投稿する本文の全文**を提示する。suggestion ブロックがあればそれも含める。これが版ズレに気づくための関門で、表＝索引、プレビュー＝全文確認の役割分担になる。

   ```
   ### 投稿プレビュー（全文）

   #### 1. path/to/A.kt:42 [インライン]
   {md の該当本文そのまま}

   #### 2. path/to/B.kt [PRコメント(行特定不可)]
   {md の該当本文そのまま}
   ```

6. **確認文を添える**

   ```
   👆 上記の全文で投稿します。最新の md（再読込済み）と一致しているか、特にご自身で編集した箇所を確認してください。
   ```

7. **`AskUserQuestion` で承認を得る（必須）**
   次の3択を提示する（または同等の選択肢提示）。**承認を得るまで投稿 API は実行しない。**

   1. **承認して投稿**（推奨）: 表・全文プレビューのとおり投稿する。
   2. **一部修正して投稿**: 修正対象の番号と修正内容（文面の修正／行の修正／指摘の削除）を指定してもらう。適用後、**md ファイルも同じ内容に更新**し、手順1（md を `cat` でディスクから読み直す）からやり直して表・全文プレビューを再提示し、再度この承認を取る（承認が出るまでループ）。行を直した場合は投稿方式を再判定する。
   3. **非承認（投稿しない）**: 投稿せずに終了する。md ファイルは残す。

8. **投稿を実行する（承認後のみ）**
   投稿方式に応じて後述のコマンドを使う。インライン＝「新規指摘の一括投稿」または「追加の行コメント」、返信＝「未解決コメントへの返信」、`PRコメント(行特定不可)`＝「PR全体へのコメント」。

   **外部書き込み API は 1 コマンド = 1 呼び出しで実行する。** `||` や `&&` で連結しない（連結すると失敗時の挙動が読めず、重複投稿の原因になる）。複数コメントは一括投稿 API（`reviews`）で 1 回にまとめるのを基本とし、個別投稿が必要な場合も 1 件ずつ独立したコマンドで実行する。投稿が失敗したように見えても、再実行する前に `gh pr view {pr_number} --comments` 等で**既に投稿されていないかを確認**し、二重投稿を避ける。

### コメント本文のルール

md・GitHub とも**重要度ラベル（`【推奨修正】` 等）は付けない**。指摘の内容と言い回しだけでトーンは伝わるため、記号や太字ラベルはノイズになる。
md ファイルの本文をそのまま GitHub コメントとして投稿してよい。

### 新規指摘の一括投稿（主フロー）

GitHub Web UI の「レビュー開始→複数コメント→まとめて送信」と同等の投稿を行う。
最新のコミットSHAを取得してから投稿する：

```bash
# コミットSHAの取得
gh pr view {pr_number} --json headRefOid --jq '.headRefOid'
```

（投稿方式の判定〔差分ハンク内かどうか〕と suggestion 紐付け範囲の確認は、上記「投稿前の確認と承認（必須ゲート）」の手順3で実施済み。以下は承認後に実行する投稿コマンド。）

**一括投稿：**

```bash
gh api /repos/{owner}/{repo}/pulls/{pr_number}/reviews \
  --method POST \
  --input - <<'PAYLOAD'
{
  "commit_id": "{コミットSHA}",
  "body": "{ユーザーが明示的に指定したテキスト。指定がなければ空文字列}",
  "event": "COMMENT",
  "comments": [
    {
      "path": "{ファイルパス}",
      "line": {行番号},
      "side": "RIGHT",
      "body": "{コメント本文（ラベルなし）}"
    }
  ]
}
PAYLOAD
```

複数行範囲に紐付ける場合は、対象コメントに `"start_line": {開始行番号}` と `"start_side": "RIGHT"` を追加する（`line` は終了行）。

`body` フィールドはデフォルト空文字列とし、ユーザーが明示的にテキストを指定した場合のみその文言を使用する。

### 追加の行コメント（一括投稿後の個別追加）

```bash
gh api /repos/{owner}/{repo}/pulls/{pr_number}/comments \
  --method POST \
  --field body="{コメント内容}" \
  --field path="{ファイルパス}" \
  --field line={行番号} \
  --field side="RIGHT" \
  --field commit_id="{コミットSHA}"
```

### 未解決コメントへの返信

```bash
gh api /repos/{owner}/{repo}/pulls/comments/{comment_database_id}/replies \
  --method POST \
  --field body="{返信内容}"
```

### PR全体へのコメント

```bash
gh pr comment {pr_number} --body "{コメント}"
```

投稿後は投稿結果のURLをユーザーに伝える。
