# コードレビュースキル

このスキルが呼ばれたら、以下の実行フローに従ってコードレビューを実施してください。
引数としてPR番号やファイルが渡された場合はその対象をレビューし、なければ現在のブランチの差分をレビューしてください。

---

## 実行フロー

### STEP 1: 現在のブランチのPR確認

```bash
gh pr view --json number,title,url,headRefName,author
```

- PRが存在しない場合 → STEP 3へ（差分のみレビュー）
- PRが存在する場合 → STEP 2へ

### STEP 2: PRの作成者確認（自分のPRかどうか）

```bash
gh api user --jq '.login'
```

取得した `login` と STEP 1 の `author.login` を比較する。

- **自分が作成したPR**: STEP 3・4・5a のみ（未解決コメントへの対応方針のみ。新規指摘は行わない）
- **他者が作成したPR**: STEP 3・5b のみ（新規指摘のみ。STEP 4・5a はスキップ）
- **PRなし**: STEP 3・5b のみ（新規指摘のみ）

### STEP 3: コード差分の取得

```bash
# PRがある場合
gh pr diff {pr_number}

# PRがない場合
git diff origin/main...HEAD
```

### STEP 3.5: パターン検索（構文で識別できる観点の事前確認）

差分に含まれる Kotlin ファイルを対象に、LLM の注意が分散しやすいパターンを grep で機械的に抽出する。
STEP 5b の新規指摘生成時に、この検索結果を必ずインプットとして参照すること。

```bash
# 差分に含まれる Kotlin ファイルを取得
gh pr diff {pr_number} --name-only | grep "\.kt$"

# 各ファイルに対して以下を実行
grep -n "@SuppressLint\|@Suppress(" {file}
grep -n "remember {" {file}
grep -n "DialogProperties" {file}
```

各ヒットに対する判断基準:

| パターン | 確認内容 | 指摘候補になる条件 |
|---------|---------|-----------------|
| `@SuppressLint` / `@Suppress(` | 直前行に `//` コメントがあるか | コメントがない場合 → コードの品質「アノテーションのコメント」として指摘 |
| `remember {` | キーなしで外部の引数（ViewModel・コールバック等）を内部に保持しているか | 保持している場合 → Jetpack Compose「rememberキー指定」として指摘 |
| `DialogProperties` | `dismissOnBackPress` / `dismissOnClickOutside` が明示されているか | 明示なし（デフォルト）かつ旧実装（`isCancelable`）から動作が変わる場合 → 確認事項として指摘 |

### STEP 4: 未解決コメントの取得（自分が作成したPRの場合のみ）

GitHub GraphQL API で未解決のレビュースレッドを取得する：

```bash
gh api graphql -f query='
{
  repository(owner: "{owner}", name: "{repo}") {
    pullRequest(number: {pr_number}) {
      reviewThreads(first: 100) {
        nodes {
          isResolved
          id
          comments(first: 10) {
            nodes {
              id
              databaseId
              body
              author { login }
              path
              line
              originalLine
              url
            }
          }
        }
      }
    }
  }
}'
```

`isResolved: false` のスレッドのみ対象とする。
owner と repo は `gh repo view --json owner,name` で取得する。

### STEP 5: レビュー実施

**a. 未解決コメントへの対応検討**（自分が作成したPRの場合のみ）
- 各コメントの内容を読み、差分コードと照らし合わせて対応要否を判断する
- 対応方針と返信案を作成する（後述のスタイル原則に従う）

**b. 差分コードへの新規指摘生成**（他者のPR / PRなしの場合のみ）

各指摘を書く際は次の順序で進める:
1. **重要度を先に決める**（内容を書く前に）
2. 後述の表でその重要度の「使う表現」を確認し、**締め文のパターン**を頭に入れる
3. 指摘の内容を書き、確認したパターンで締める
4. 書き終えたら**締め文が表と一致しているか確認する**（推奨修正で「できないでしょうか？」が出たら書き直す）

**既存コードの引用について:**
`suggestion` ブロックは提案コードのみに使い、**既存コードはコードブロックで引用しない**。
「現在の実装は以下の通りです」と書いて既存コードを貼りたくなった場合は、コードを貼らず「〜という実装になっていますが」と言葉だけで状況を説明する。
レビュアーはすでにコードを手元で見ているため、既存コードの引用はノイズになる。

- 後述のレビュー観点チェックリストに従い指摘を生成する
- 各指摘には **ファイル** と **行** を必ず記載する（行が特定できない場合は `-`）
  - `gh pr diff` の出力はdiff上の行番号であり、実際のファイルの行番号と異なる。**実際のファイル行番号**を取得するには `grep -n "{特徴的なコード}" {ファイルパス}` で確認する
  - 行番号を誤って記載するとレビュアーが該当箇所を探せないため、確認できない場合は `-` にする
- **指摘事項は重要度順（必須修正→推奨修正→確認事項→任意修正）で出力する**
- 重要度ごとの言い回しは以下のルールに従う。

| 重要度 | 使う表現（具体例） | 禁止表現（絶対使わない・具体例） |
|--------|----------|--------------------------|
| 必須修正 | 「〜に変更できないでしょうか？」「〜に修正できないでしょうか？」（**締め文は必ずこの形で終える**。「〜はいかがでしょうか？」で終えたくなった場合は重要度を推奨修正に変更すること） | 「〜していただけますでしょうか？」「ご確認いただけますでしょうか？」「確認させてください」「ご確認のうえ〜」「〜はいかがでしょうか？」（これは推奨修正の表現） |
| 推奨修正 | 「〜はいかがでしょうか？」「〜という形はどうでしょう？」（**締め文は必ずこの形で終える**。確認事項の表現で終えたくなった場合は重要度を確認事項に変更すること） | 「〜できないでしょうか？」（**動詞の種類・形を問わず「〜できないでしょうか？」はすべて禁止**。「変更できないでしょうか？」「統一できないでしょうか？」「抽出できないでしょうか？」「分けられないでしょうか？」「まとめられないでしょうか？」「移動できないでしょうか？」など形が変わっても同様に禁止。推奨修正は「提案についてどう思うか」を問うので「いかがでしょうか？」を使う）「〜していただけますでしょうか？」「意図的な設計でしょうか？」「意図はございますでしょうか？」（これらは確認事項の表現） |
| 任意修正 | 「スルーでも問題ありませんが〜」「このままでも問題ありませんが〜」で**必ず書き始める** | 「〜修正していただけますでしょうか？」「〜できないでしょうか？」 |
| 確認のみ | 「意図的な設計でしょうか？」「〜という意図はございますでしょうか？」「〜という理解でよろしいでしょうか？」（**文末は必ず「〜でしょうか？」の疑問文で終える**） | 「〜できないでしょうか？」「確認できないでしょうか？」（「〜できないでしょうか？」の形はすべて禁止）「〜していただけますでしょうか？」「確認させてください」「〜確認しておきたいのですが」「〜ください」「〜お願いします」（確認事項は相手に行動を求めるのではなく、意図・理解を問う純粋な疑問文にする） |

> **重要**: この言い回しルールは指摘の締め文だけでなく、**指摘内容の本文（改善案・代替案の提示部分）にも適用する**。
> - 確認事項の本文内に改善案を添える場合は「〜という方法も考えられます」「〜という形も一案ですが」にとどめ、「〜できないでしょうか？」は使わない。
> - 「確認できないでしょうか？」は「〜できないでしょうか？」の一形態であり、確認事項・推奨修正・任意修正での使用は**禁止**。必須修正でのみ使用可能だが、その場合も「確認」ではなく「修正・変更」を求める動詞を使う。

**各重要度の概念的な違い（表現選択に迷ったときの判断基準）:**
- **必須修正** → 「この問題を修正することは可能ですか？（可否を問う）」→「〜できないでしょうか？」
- **推奨修正** → 「この提案についてどう思いますか？（意見を問う）」→「〜はいかがでしょうか？」
- **任意修正** → 「問題はないが、もし良ければ（提案を先に否定しない）」→「このままでも問題ありませんが〜」で始める
- **確認事項** → 「これは意図的ですか？（意図・理解を問う）」→「〜でしょうか？」の疑問文

推奨修正で「〜できないでしょうか？」が浮かんだら、それは必須修正か確認事項の表現なので重要度の見直しを検討すること。

### STEP 5.5: 出力前セルフチェック

mdファイルを書く前に、生成した各指摘を次の観点で確認する:

- [ ] 推奨修正の指摘が「〜はいかがでしょうか？」「〜どうでしょう？」で終わっている（「できないでしょうか？」が入っていたら書き直す）
- [ ] 既存コードをコードブロックに引用していない（`suggestion` ブロックは提案コードのみ）
- [ ] 確認事項の指摘が「〜でしょうか？」の疑問文で終わっている（「確認させてください」「〜ください」は書き直す）
- [ ] 任意修正の指摘が「このままでも問題ありませんが」または「スルーでも問題ありませんが」で始まっている
- [ ] 必須修正・推奨修正・確認事項の指摘内容の最初の文が「？」で終わっている（「〜です。」「〜ます。」「〜います。」で始まっていたら書き直す）

### STEP 6: mdファイルの生成・配置

```bash
mkdir -p {カレントディレクトリ}/tmp/review
```

ファイル名: `review_{pr_number}_{title}.md`（PRなしの場合は `review_branch_{YYYYMMDD_HHMMSS}.md`）

`{title}` は `gh pr view --json title --jq '.title'` で取得したPRタイトル文字列をそのまま使う。
ファイル名に使えない文字（`/`・macOS で問題になる `:` など）は `_` に置換する。

後述の出力フォーマットに従いmdファイルを作成し配置する。

### STEP 7: 結果の報告

- 作成したmdファイルのパスをユーザーに伝える
- 未解決コメントがあった場合は件数と対応要否の内訳を簡潔に伝える
- 「コメントを投稿する場合は指示してください」と添える

## レビュースタイルの原則

このスタイルは **新規指摘コメント** と **未解決コメントへの返信案** の両方に適用する。

1. **優先度に応じた言い回しを使い分ける**
   - 実行を求める場合は「〜していただけますでしょうか？」（実行有無を問う）ではなく「〜できないでしょうか？」（実行可否を問う）を使う
   - 相手が何らかの理由でできない可能性がある場合、実行有無を問うと主張が衝突するリスクがある
   - 推奨・任意・確認の指摘: 断定せず「〜はいかがでしょうか？」「〜という背景はございますでしょうか？」と問いかける
   - 自分の理解が正しいか確認する質問と、改善提案を明確に区別する

2. **「提案→理由」の順で書く**
   - 構成の基本形: まず提案・質問を書き、その後に理由・背景を1文で添える
   - **現状の実装を説明する文（「〜という実装になっています」「〜が〜されています」「〜状態です」）は最初に書かない**。コードを見れば分かる情報であり、理由の説明で補完できる。

   **推奨修正の例:**
   - NG: 「`Content.Launched` と `Content.OnLoad` で同じ引数列が2か所に重複しています。フィールドを追加したときに片方が漏れやすいので、共通関数に切り出す形はいかがでしょうか？」
   - OK: 「共通関数に切り出す形はいかがでしょうか？`Content.Launched` と `Content.OnLoad` で同じ引数列が2か所に重複しており、フィールドを追加したときに片方が漏れやすいためです。」

   **確認事項の例（現状説明を疑問文に折り込む）:**
   - NG: 「`dismissOnBackPress` がデフォルト（true）になっているため、バックキーでダイアログを閉じられる状態です。旧実装の `isCancelable = false` から意図的に変更したのでしょうか？」
   - OK: 「`isCancelable = false` を外して `dismissOnBackPress` をデフォルト（true）にしたのは意図的でしょうか？」（「何を変えたか」を疑問文の主語にする）
   - NG: 「`remember` にキーが渡されていないため、引数が変わっても再生成されません。意図的な設計でしょうか？」
   - OK: 「`remember` にキーを渡さない設計は意図的でしょうか？引数が変わっても再生成されないためです。」

   - 理由がコードスニペットや質問文で伝わる場合は省略してよい

3. **実装案・代替案を提示する**
   - コードスニペット（suggestionブロック）は提案コードのみに使う。既存コードはソースコードを見れば分かるため引用しない
   - typoなど受け取り手が自己解決できる指摘は、修正案を書かずに指摘のみにする
   - 提案は最小限の修正を優先する。大規模な設計変更は「〜という方法も考えられますがいかがでしょうか？」と軽く触れる程度にとどめる

4. **修正の強制度を明示する**
   - 必須修正：「〜できないでしょうか？」
   - 任意修正：「このままでも問題ありません」「軽微なのでスルーでも問題ありませんが」
   - 確認だけ：「〜という背景はどのような場合でしょうか？」

5. **細かい指摘には謙遜を添える**
   - 「めっちゃ細かいのですが」「細かくて恐縮ですが」など、些細な指摘であることを先に断る
   - 自分の理解が浅い可能性を認め、「私の理解が浅くてご質問です」と添える

6. **相手に確認を促す**
   - 判断が難しい場合は「〜と思うのですがいかがでしょうか？」「〜はどうでしょうか？」と問いかける
   - 返信案でも自分の方針・意図・背景を示したうえで、「〜という認識でよろしいでしょうか？」「いかがでしょうか？」と添える

7. **誤解・ミスを素直に認める**
   - 自分の指摘が間違いだった場合は素直に認め、謝罪する

8. **文章は簡潔に書く**
   - 説明口調・冗長な言い回しを避ける
   - 背景や理由は添えるが、一文で収まる程度に簡潔にする
   - 「1点目・2点目」のような番号付き構造化はしない。1指摘につき観点は1つに絞り、複数の懸念がある場合は重要度の高いものだけを指摘する

9. **「〜べき」という表現は使わない**
   - 背景や意図を確認していない状態で断言できないため
   - 代わりに「〜はいかがでしょうか？」「〜という形にするのはどうでしょう」と提案形にする

10. **複数の文を続ける場合は文ごとに改行する**
    - 一文ごとに改行し、詰め込みすぎない

11. **断定せず、可能性の提示をする**
    - 「〜かもしれません」「〜ような気がします」「〜と思われます」などの表現を使う
    - 自分の判断が誤っている可能性を常に考慮する

12. **一度記載した情報は繰り返さない**
    - 指摘の説明内で同じ内容を再掲しない

## 未解決コメントへの対応方針の判断基準

| 判断 | 内容 |
|------|------|
| **対応必要** | コメントの指摘が妥当で、差分コードに修正の余地がある |
| **対応不要** | すでに修正済み、仕様上意図的な実装、またはコメントの前提が誤っている |
| **要確認** | コメントの意図が不明確、または判断が難しい |

## レビュー観点チェックリスト

### 1. 命名規則

- **typoをチェックする**
  - スペルミスを「typoかもしれません」と確認形式で指摘する
  - コメント文中のtypoも対象にする

- **命名の具体性・適切な抽象度**
  - `data`・`info`・`flag` など広すぎる（汎用すぎる）名前は具体化を促す
  - 特定の用途に限定されすぎる場合は汎用化を提案する

- **プロジェクト内の命名の一貫性**
  - 既存コードの命名規則に合わせることを求める
  - 同じ概念に複数の命名が混在している場合は統一を求める（例：`EKYC, Ekyc, EKyc` の混在）

- **実態に合った命名**
  - メソッド名と処理内容が一致しているか確認する（例：fetchしているだけなのに `update` という名前）
  - スコープを明確にする命名（例：`QuickDepositBank` のようにコンテキストを含める）

- **Boolean の命名**
  - Boolean 変数・プロパティは `is/has/can` 等で始まっているか確認する

- **その他確認ポイント**
  - `Author` 情報など不要なコメントが残っていないか

---

### 2. コードの可読性

- **trailing comma の追加**
  - Kotlin のコーディング規約に従い、関数の引数・コレクション要素の末尾カンマの追加を促す

- **ネストを減らす・早期リターンを使う**
  - 例：`val userId = userIdUseCase.get().first() ?: return`

- **肯定を先に処理する**
  - `isAllValid` のようなメソッド名なら内部ロジックも肯定を軸に統一する

- **スコープ関数の適切な使い分け**
  - null チェックは `?.let { … }` で書き換えるとシンプルになる場合を指摘する
  - `let/also/apply/run/with` を目的に合わせて使い分けているか確認する

- **ラムダ引数の命名**
  - ラムダ内で `it` を使うと文脈が失われる場合は、意味のある名前への変更を提案する（例：`forEach { item -> … }`）

- **`val` の積極利用**
  - 再代入しない変数は `var` ではなく `val` で宣言できないか確認する
  - `val` になった場合に `get()` を省略できるか確認する

- **冗長なコードを整理する**
  - ほぼ同じ処理が複数箇所に定義されている場合は統合を提案する
  - `private val` が init でしか使われていない場合は引数への変更を提案する
  - コメントアウトされたままのコードが残っていないか確認する

- **意図せず含まれた差分を確認する**
  - このPRのスコープと無関係なファイルの変更が含まれていないか確認する
  - `.idea/` 等の IDE 設定ファイルが混入していないか確認する

---

### 3. 設計・アーキテクチャ

- **責務の分離**
  - 画面ロジック（条件分岐）が Composable に含まれている場合は ViewModel 側に移動を促す
  - ViewModel から Repository を直接呼ばず UseCase を経由しているか確認する
  - ScreenState はデータのみを持ち、メソッド・ビジネスロジックを持たないか確認する
  - バリデーション結果（Valid/Invalid）は ViewModel または UseCase 側に持たせ、画面側に持たせない

- **共通化の判断基準（YAGNI原則）**
  - 再利用が見込めるか、共通化によって設計思想が分かりにくくならないか
  - 「事前に予測した自由度で設計を複雑にするより、要件が出た時に対応する方が良い」（YAGNI原則）
  - Composable 関数が大きくなった場合は適切に分割しているか確認する

- **Widgetのパディング設計**
  - パーツ側はパディングを持たず、利用側で付けてもらう設計が推奨
  - 上下左右のパディングが不揃い（「左右にはあるが下だけない」等）になっていないか確認する

- **デッドコードの除去**
  - どこからも呼び出しがないコードは積極的に削除するか指摘する
  - 未使用の import・フィールド・メソッドを確認する

---

### 4. コルーチン・Flow 設計

- **StateFlow / SharedFlow の使い分け**
  - 画面 UI 状態は `StateFlow` で管理しているか
  - 画面遷移・ダイアログ表示など一度だけ発行すべきイベントは `SharedFlow` で管理しているか
  - `StateFlow` の初期値は適切か（`null` が不要な場合は `Unit` や適切なデフォルト値にする）

- **UiEvent の設計**
  - UiEvent は画面ごとの sealed interface/class に集約されているか
  - イベント消費（consume）の順序：処理を実行してから消費する
  - 同一 UiEvent の複数箇所で消費タイミングが異なる場合は統一を促す

- **スレッド・Dispatcher**
  - `runBlocking` をメインスレッドで使っていないか（ANR リスク）
  - DB 操作・ネットワーク通信は適切な Dispatcher（`IO` 等）で行っているか
  - `viewModelScope.launch` でエラーハンドリングができているか

- **Flow の合成**
  - `combine` / `flatMapLatest` ブロック内でエラーハンドリングが漏れていないか（例：一方の Flow が常に Success を返す場合、エラーが伝播しないリスク）

- **ViewModel / UseCase**
  - 画面ログ（Screen Log）は ViewModel の `init` で送信する設計方針に準拠しているか
  - `UseCase` 命名に HTTP メソッド名（`Get`, `Post`）が含まれていないか（ドメイン観点の命名が適切）
  - ViewModel のプロパティ（Logger 等）が `private` でなく外部から直接参照できる状態になっていないか
  - 画面ログ・分析ログ（`screenEvent` / `tapEvent` 等）の削除・変更が含まれている場合、意図的かどうかを確認する

---

### 5. Jetpack Compose

- **状態の収集**
  - `collectAsState` ではなく `collectAsStateWithLifecycle` を使っているか
  - UiEvent のコレクトは `repeatOnLifecycle(STARTED)` （または `LaunchedEffect` + `repeatOnLifecycle`）で行っているか

- **LaunchedEffect**
  - Composable 本体（コンポーズフェーズ）で `uiEvents.forEach { ... }` のように副作用を直接実行していないか。副作用は Effect API で実行する
  - `forEach` ブロック内で `return` を使うと非ローカルリターンになり後続のイベント処理がスキップされる。`return@forEach` が必要なケースか確認する
  - `LaunchedEffect` のキーは適切か（`Unit` にして良いケース vs 可変値が必要なケース）

- **remember**
  - `remember { ... }` にキーが指定されていない場合、引数が変わっても再生成されない。`remember(key) { ... }` が必要かどうかを確認する
  - `LocalContext.current` を `remember` キーなしの安定クラス内で保持している場合、Activity 再生成後に古い Context を参照するリスクがある

- **Preview**
  - Preview が追加・更新されているか、他の Composable と一貫した内容になっているか
  - `@Preview(showBackground = true)` の付与を確認する
  - Fake 実装クラスは Preview メソッド内に定義する方がわかりやすい

- **その他**
  - `Modifier.fillMaxWidth()` でタップ領域を画面端まで広げているか（テキストリンク等）
  - `DialogProperties` で `dismissOnBackPress` / `dismissOnClickOutside` をデフォルトのまま使っている場合、旧実装からの動作変更が意図的かどうかを確認する

---

### 6. コードの品質

- **null 安全の扱い**
  - `!!`（non-null assertion）を使っていないか
  - `null` を引数に渡すことで実質「何もしない」処理になっていないか確認する
  - Kotlin の `?.let` や `?: return` で null チェックを簡潔に書く

- **アクセス修飾子**
  - 外部からアクセスされないメンバーは `private` / `internal` にしているか
  - ViewModel のプロパティが不必要に公開されていないか（`_uiState` と `uiState` のパターン等）

- **定数・マジックナンバー**
  - ハードコードされた数値・文字列は `const val` または `companion object` で定数化しているか
  - マジックナンバー・マジックストリングを使っていないか

- **バリデーション・境界値**
  - 条件の `<` と `<=` の正確性を確認する
  - 空リスト・空文字・境界値の入力に対応しているか

- **文字列リソースの扱い**
  - 既存の文字列定義（`strings.xml`）と同じ文言を直書きしている場合は既存定義の使用を促す

- **アノテーションのコメント**
  - `@SuppressLint` / `@Suppress` でワーニングを抑制している場合、なぜ抑制しているかのコメントが書かれているか確認する

---

### 7. コメント・ドキュメント

- **WHY を書く（理由・背景の記述）**
  - 「何をしたか」の事実ではなく「なぜそうしたか」の根拠を書くことを強く求める
  - コードを読めばわかる結果ではなく、コードからは読み取れない背景を書くよう促す
  - 処理の順序制約・ライフサイクルの制約がある場合はその理由をコメントしているか

- **実態との乖離を防ぐ**
  - コメントの内容がコードの実態と合っているか確認する
  - メソッドを大きく書き換えた場合はコメントやメソッド名も更新を促す

- **未来の開発者への配慮**
  - 特殊な処理は、消してしまうと問題が起きることを説明するコメントを残す

- **TODOコメント**
  - 今回対応できない部分は TODO コメントとして残すことを提案する
  - TODO には担当者・チケット番号を記載するよう促す

- **AIへの指示の明確化（Copilot / Claude 対象）**
  - 曖昧さがある場合は具体化を求める（「適切」のような抽象的な表現を避ける）
  - 原則に例外がある場合は例外を明記する
  - チーム全体に適用すべきルールであれば Copilot Instructions への追記を検討する

---

### 8. セキュリティ・堅牢性

- **エラーハンドリング**
  - 予期しない API レスポンス（null・空リスト・エラーステータス）のハンドリングができているか
  - 例外を `catch` している場合、Crashlytics にエラーを記録しているか

- **ログへの秘密情報出力**
  - ログにトークン・個人情報・認証情報を出力していないか

- **データ整合性**
  - 複数データの更新はアトミックに行っているか（トランザクション・整合性）
  - Room Migration のスキーマファイル（`schemas/*.json`）が更新されているか

---

### 9. テスト・品質担保

- **テストコードとプロダクトコードの整合**
  - 変更したロジックに対応するテストを追加・更新しているか
  - テストのモック・スタブが実際の API レスポンスと整合しているか

- **テストケースの質**
  - テストケースが重複していないか確認する
  - 境界値・エラーケース・エッジケースのテストがあるか

- **動作確認エビデンス**
  - UI の変更がある場合、動作確認のスクリーンショット（または GIF）を PR 説明に添付しているか
  - Before/After の確認が必要な変更は両方のスクリーンショットを用意しているか
  - リリースビルド（Proguard/R8 適用）での動作確認が必要なケースは事前確認を促す

## レビュー時の優先度判断

| 優先度 | 内容 | 対応例 |
|--------|------|--------|
| 高（必須修正） | バグ・仕様漏れ・クラッシュリスク | 「〜できないでしょうか？」 |
| 中（推奨修正） | 可読性・命名・設計一貫性 | 「いかがでしょうか？」 |
| 低（任意） | スタイル・細かいリファクタリング | 「このままでも問題ありません」 |
| 確認のみ | 実装の意図・背景の理解 | 「〜という背景をお伺いできますでしょうか？」 |

## PRへのコメント投稿

ユーザーから「投稿して」「コメントして」「返信して」などの指示があった場合のみ実施する。

### コメント本文のルール

md ファイル内では重要度ラベル（`【推奨修正】` 等）を使うが、**GitHub に投稿する際はラベルを除いた本文のみを使用する**。
指摘の内容と言い回しだけで重要度は伝わるため、記号や太字ラベルはノイズになる。

### 新規指摘の一括投稿（主フロー）

GitHub Web UI の「レビュー開始→複数コメント→まとめて送信」と同等の投稿を行う。
最新のコミットSHAを取得してから投稿する：

```bash
# コミットSHAの取得
gh pr view {pr_number} --json headRefOid --jq '.headRefOid'
```

**投稿前に差分外の行がないか確認する：**

```bash
gh pr diff {pr_number} | grep "^@@"
```

GitHub REST API の行コメントは差分ハンクに含まれる行（変更行＋前後のコンテキスト行）にしか投稿できない。
差分外の行が含まれる場合は投稿前にユーザーへ通知し、以下の対処を提示する：
- **PR 全体コメントとして投稿**（行の特定はできないが内容は伝わる）
- **近傍の変更行に移動して投稿**
- **投稿をスキップ**（任意修正など優先度が低い場合）

**一括投稿：**

```bash
gh api /repos/{owner}/{repo}/pulls/{pr_number}/reviews \
  --method POST \
  --input - <<'PAYLOAD'
{
  "commit_id": "{コミットSHA}",
  "body": "{ユーザーが明示的に指定したテキスト。指定がなければ空文字列}",
  "event": "COMMENT",
  "comments": [
    {
      "path": "{ファイルパス}",
      "line": {行番号},
      "side": "RIGHT",
      "body": "{コメント本文（ラベルなし）}"
    }
  ]
}
PAYLOAD
```

`body` フィールドはデフォルト空文字列とし、ユーザーが明示的にテキストを指定した場合のみその文言を使用する。

### 追加の行コメント（一括投稿後の個別追加）

```bash
gh api /repos/{owner}/{repo}/pulls/{pr_number}/comments \
  --method POST \
  --field body="{コメント内容}" \
  --field path="{ファイルパス}" \
  --field line={行番号} \
  --field side="RIGHT" \
  --field commit_id="{コミットSHA}"
```

### 未解決コメントへの返信

```bash
gh api /repos/{owner}/{repo}/pulls/comments/{comment_database_id}/replies \
  --method POST \
  --field body="{返信内容}"
```

### PR全体へのコメント

```bash
gh pr comment {pr_number} --body "{コメント}"
```

投稿後は投稿結果のURLをユーザーに伝える。
