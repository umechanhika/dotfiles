# pre-commit-leak-check

ユーザーが Claude に commit / push / PR 作成操作を依頼したら、実行する前にステージ済み差分をレビューし、機密情報・個人情報・業界特定情報・端末固有情報が混入していないかを確認する。**コミット対象のファイル一覧と各ファイルの4観点チェック結果を表にまとめて提示し、ユーザーが明示的に承認するまで commit しない。** 承認を得たら commit し、続けて push する（upstream 未設定なら `-u` でローカルブランチを upstream に設定）。承認時に**どこまで実行するか（push まで / PR 作成まで）をユーザーが選び**、`PR 作成まで` が選ばれた場合は push 後に PR を作成する。

機密情報の検出が 0 件であっても、承認ゲートは必ず通す。チェックが「機密的に OK」でも、ユーザーがそもそもコミットしたくないファイル（誤って `git add` した実験/一時/無関係な変更）が紛れている可能性があり、それを人間の目で止められるようにするためである。表でファイル単位の全体像を見せることで、機密の有無とは独立に「このファイルは入れたくない」という判断ができる。

## なぜこのチェックが必要か

dotfiles や設定ファイルは公開リポジトリに置かれることが多く、過去にも「コメントの例示を汎用的なパス表記に修正」のような後追い修正が発生している。業務リポジトリでも、ローカル絶対パス・社内 URL・実名メールアドレス等が誤って混入すると、PR レビュー時の指摘やリビジョン履歴の汚染につながる。コミット直前に観点別レビューを挟むことで、後追い修正の発生を抑えられる。

## スクリプトと LLM の役割分担

機械的に決定論で処理できる箇所はバンドルスクリプトに委譲し、コンテキスト消費を抑える。LLM は**意味的な判断（A〜D レビュー）と承認対話**に集中する。

- `${CLAUDE_SKILL_DIR}/scripts/scan-staged.sh` … コミット対象一覧（増減/バイナリ/サイズ）と機械的な漏えいシグナル（端末パス・秘密鍵/トークン署名・メール・IP を実ファイル行番号付きで）を 1 呼び出しで収集。**補助シグナルであり、意味的判断は LLM が行う。**
- `${CLAUDE_SKILL_DIR}/scripts/publish.sh` … push（remote 決定・upstream 判定・`-u` フォールバック）と PR 作成（GitHub 解決判定・base 検出・既存 PR 確認・gh アカウント切替のアトミック実行・PR 作成）を 1 呼び出しで実行。

両スクリプトとも想定外は黙って継続せず非ゼロ終了する（フォールバック禁止）。出力やエラーはそのままユーザーに提示し、勝手なリトライ・回避はしない。

## スキップ条件

ユーザーが「チェック不要でコミットして」「漏えいチェックなしで」「機密チェックスキップ」のように**明示的に**指示した場合に限り、本スキルをスキップして直接 commit に進んでよい。単に「コミットして」と言われたケースはすべて実施する。

## 実行フロー

### STEP 1: ステージ状況の確認 + 機械的スキャン

```bash
bash "${CLAUDE_SKILL_DIR}/scripts/scan-staged.sh"
```

- このスクリプトはステージ済み差分が空なら非ゼロ終了する。その場合は「ステージされている変更がありません。`git add` で対象ファイルを指定してください」と伝えて中断する。
- 出力の `## staged files` がコミット対象一覧（STEP 3 の表の素データ）。バイナリ・大容量ファイルはサイズが併記されるので、意図したものか確認する。
- 出力の `## mechanical hits` は正規表現で拾った**補助シグナル**（カテゴリ・実ファイル行番号・マスク済みスニペット）。見落とし防止のための材料であって、これだけで判断しない。
- ステージ漏れがありそうな場合（`git status` に未ステージ変更がある）は、commit に含めるか確認する。必要なら `git status --short` を併用する。

### STEP 2: 4 観点での意味的レビュー

scan のシグナルを踏まえたうえで、**差分本文を必ず目視で読む**:

```bash
git diff --staged
```

正規表現では拾えない文脈依存の漏えい（社内コードネーム、業界特定の用語の組み合わせ等）は LLM の判断でしか検出できない。scan のヒットは出発点として使い、差分全体を A〜D の観点で読む。リスクの高いもの（API キー・秘密鍵・実名/業務メール）は最優先で抽出する。

#### A. 企業の機密情報

- 実在する企業名・プロダクト名（特に未公開のもの、社内コードネーム）
- 社内システム名・内部ツール名・社内ドメイン
- 顧客名・取引先名
- 内部 URL（社内 Confluence / Jira / Slack / Grafana など）
- 未公開の財務情報・KPI 数値・売上規模

#### B. 個人を特定できる情報

- 実名のフルネーム（自分・同僚問わず）
- 業務用メールアドレス（ドメインから業界が推測されるものを含む）
- 個人メールアドレス、電話番号、SNS アカウント、社員 ID
- 「user's email/name」のようにユーザー情報を保持しているメモリやコンテキストが差分に貼り付いていないか

ユーザー本人の実名や業務メールでも、公開リポジトリに残れば業界が特定されるリスクがある。検出したらフラグを立てる。

#### C. 業界・業種を特定できる内容

- 特定業界の専門用語の組み合わせ（例：金融商品名・医療コード・防衛関連略語・暗号資産プロダクト名・SaaS 製品名など）
- 業界固有の規制名・サービス名・商品名
- 業界文脈を含むサンプルデータ・コメント（テーブル名・カラム名・列挙値）

単独では問題ない単語でも、組み合わせで業界推定が容易になる場合は指摘する。

#### D. 端末・環境固有情報

- 絶対パス（`/Users/<実名>/...`、`/home/<実名>/...`、ホスト名を含むパス）
- ホスト名・ローカル IP・社内ネットワークアドレス
- API キー・アクセストークン・OAuth トークン・秘密鍵（`-----BEGIN ... PRIVATE KEY-----`、`AKIA...`、`ghp_...`、`xoxb-...` など）
- `.env` の生値・データベース接続文字列・パスワード
- シェル履歴・ローカル設定ファイル（macOS の `.DS_Store` など）

API キー・秘密鍵・トークン類は **検出時点で commit を止め、修正後の再チェックを必須とする**。漏えいすると即時のローテーションが必要になるため、リスクが他カテゴリと桁違いに高い。

### STEP 3: コミット対象とチェック結果の提示

**検出の有無にかかわらず、まず必ず以下の表を提示する。** 表には scan 出力の `## staged files` を **1 件残らず** 列挙する（機密検出が無いファイルも省略しない）。これは、機密の有無とは別に「そもそもコミットしたくないファイルが混ざっていないか」をユーザーがファイル単位で確認できるようにするためである。

```
## コミット対象とチェック結果

| パス | 変更概要 | A 企業機密 | B 個人情報 | C 業界特定 | D 端末固有 | 総合 |
|------|---------|-----------|-----------|-----------|-----------|------|
| path/to/file1 | 設定追加 (+12/-0) | OK | OK | OK | OK | ✅ OK |
| path/to/file2 | パス修正 (+3/-3) | OK | OK | OK | ⚠️検出 | ⚠️ 要確認 |
```

表の書き方:

- **「変更概要」** は差分の要旨と増減行数（scan 出力の `+追加 -削除`）を簡潔に。
- 各観点セルは `OK` か `⚠️検出` のいずれか。1 つでも `⚠️検出` がある行は総合を `⚠️ 要確認`、すべて OK の行は `✅ OK`。
- バイナリ・大容量ファイルは「変更概要」にサイズを併記し、意図したものか目立たせる。

表の直後に、次の確認文を **必ず** 添える:

```
👆 この中にコミットしたくないファイルが含まれていないか確認してください。
不要なファイルがあれば指定してください。`git restore --staged <file>` で個別に unstage します。
```

機密検出が 1 件以上ある場合は、表の **後に** 続けて下記の詳細フォーマットで該当項目を列挙する（検出 0 件なら詳細パートは不要で、そのまま STEP 4 の承認へ進む）。

```
機密性チェック結果: 検出 N 件

## 高リスク（即時対応推奨）
※ API キー・秘密鍵・トークンは漏えいすると即時ローテーションが必要なため、ここに最優先で列挙する。

### [D] path/to/file:行番号
**検出**: `AKIA****************`
**リスク**: AWS アクセスキーがコミットに混入。漏えい時はキーローテーションが必要。
**修正案**: 該当行を削除し、環境変数 `AWS_ACCESS_KEY_ID` から参照する形に変更。

## カテゴリ別

### [A] 企業の機密情報
- path/to/file:12 `<社内ツール名>` → 「内部ツール」など汎用表記に置換

### [B] 個人を特定できる情報
- path/to/file:5 `user.name@example.com` → コミット用メール（`noreply@...`）または `user@example.com` に置換

### [C] 業界・業種を特定できる内容
- ...

### [D] 端末・環境固有情報
- path/to/file:3 `/Users/<your-username>/work/...` → `~/work/...` または `<project-root>/...` に置換
```

記述のルール:

- **検出箇所はファイル名と実ファイルの行番号で示す**（scan 出力の行番号は実ファイル行）。特定できない場合は `-`。
- **該当行の抜粋は最大 80 字程度**にトリミング。秘密鍵などは全文を貼らずマスクする（`AKIA****************`）。
- **修正案は具体的に書く**。「適切に直してください」のような曖昧表現は避け、置換後の文字列例まで示す。
- **「削除推奨」「マスク推奨」「汎用化推奨」など処置タイプを明示**する。
- pre-code-review スキルの言い回しに揃え、「〜できないでしょうか？」「〜はいかがでしょうか？」を使い分ける（必須修正と推奨修正で語尾を変える）。

### STEP 4: ユーザー承認の確認（全ケース必須）

**この承認ゲートは検出 0 件でも必ず通す。承認を得るまで commit / push / PR 作成は実行しない。** これがこのスキルの中核となる不変条件である。表とコミットメッセージ案を提示したうえで、`AskUserQuestion` ツールで次のアクションを選んでもらう。

**どこまで作業するか（push まで / PR 作成まで）も承認時にユーザーが選ぶ。** リポジトリや作業内容によって変わるため、範囲のどちらにも「推奨」を付けず、毎回フラットに選んでもらう。

承認を求める際は、コミットメッセージ案も併せて提示する:

```
コミットメッセージ案:
{type}: {内容}

承認後、このメッセージで commit し、選ばれた範囲まで実行します
（upstream 未設定のブランチは -u で upstream を設定）。
```

選択肢（状況に応じて提示する）:

1. **承認 — commit → push → PR 作成**: 表のとおりで問題なく、PR まで作りたい場合。**この選択をもって PR 作成の承認とみなし、作成直前の追加承認は挟まない**（base / タイトル / 本文は自動生成。内容はブラウザで確認・編集できる）。
2. **承認 — commit → push（PR は作らない）**: push までで止めたい場合。
3. **ファイルを除外 / 検出箇所を修正してやり直す**: STEP 5 で個別 unstage / Edit→再ステージ→再チェックを行い、表を再提示する（除外と修正は併用可）。
4. **中止**: スキルを終了する。

PR 肢（選択肢 1）の出し分け:

- 選択肢 1 は **PR を作成できる環境のときだけ提示する**。判定は `gh repo view --json nameWithOwner` が成功するか（gh が対象 remote を GitHub リポジトリとして解決できるか）で行う。**URL に `github.com` を含むかの文字列一致では判定しない**——`git@github-xxx:owner/repo.git` のような SSH ホストエイリアスを取りこぼすため。`gh` 未認証や GitHub 以外で失敗するなら選択肢 1 を出さない。
- 書き込み権限は publish.sh が確認・調整する（READ でも write 権限を持つ別アカウントへ自動で一時切替して作成しうる）ため、ここでは権限まで判定しない。
- 選択肢 1 を出さない場合は「この remote では PR を作成できないため、実行は push までになります」と一言添える。満たさない環境で勝手に push までへ切り替えて黙って進めない。

注意:

- 機密検出（特に API キー・秘密鍵・トークンなどの高リスク）がある状態で選択肢 1 または 2（push を伴う実行）を選ぼうとしたら、「これは公開しても問題ない値ですか？」と確認し、そのまま commit してよいかを念押しする。高リスク検出はローテーションが必要になるため、原則は選択肢 3 で修正してもらう。
- 選択肢 3 では不要ファイルの除外と機密修正を併用しうる。その場合は両方こなしてから STEP 4 をやり直す。

### STEP 5: 除外・修正と再チェック

選択肢 3 を選ばれたら、まず「ファイルを除外するか / 検出箇所を修正するか（両方可）」を確認し、該当する手順を行う。

**ファイルを除外する場合**:

1. 指定ファイルを **個別に** `git restore --staged <file>` で unstage する。`git reset`（全解除）は他の意図したステージを巻き込むため使わない。
2. STEP 1（scan 再実行）に戻り、表を再生成して再提示 → STEP 4 で改めて承認。

**機密検出を修正する場合**:

1. 指摘箇所を Edit ツールで修正する。複数ファイルにまたがる場合は変更ごとに分けて Edit する。
2. 修正対象ファイルを **個別に** `git add <file>` で再ステージする。`git add -A` / `git add .` は意図しないファイルを巻き込むため使わない。
3. STEP 1（scan 再実行）に戻り、修正反映・新規問題混入を確認 → 表を再提示 → STEP 4 で改めて承認。残存問題があればループする。

いずれも、**承認（選択肢 1 または 2）が得られて初めて STEP 6 に進む**。

### STEP 6: commit → publish.sh で push →（任意で）PR 作成

承認（選択肢 1 または 2）が得られたら、まず commit し、続けて publish.sh で push（と PR）を行う。

#### 6-1. commit

システムプロンプトの git commit 規約に従って commit する。要点のみ再掲：

- HEREDOC でメッセージを渡す。
- `Co-Authored-By` 行を末尾に追加する。
- `--amend` は使わず、新規 commit を作る（pre-commit フック失敗時も amend ではなく原因修正→新規 commit）。
- `--no-verify` でフックをスキップしない。
- commit 後に `git status` で結果を確認する。

#### 6-2. push（と PR）— publish.sh

remote 決定・upstream 判定・push・PR 作成は publish.sh に委譲する。承認された範囲で `--mode` を切り替える。

**push まで（選択肢 2）**:

```bash
bash "${CLAUDE_SKILL_DIR}/scripts/publish.sh" --mode push
```

**PR 作成まで（選択肢 1）**: PR 本文を一時ファイルに用意してから渡す。

```bash
# タイトル: コミット 1 件ならその subject、複数なら要約 1 行。
# 本文: 変更概要（ファイル別の要点・背景）。末尾にシステムプロンプトの PR 規約フッターを必ず含める。
BODY=$(mktemp)
cat > "$BODY" <<'EOF'
## 概要
<変更概要>

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
bash "${CLAUDE_SKILL_DIR}/scripts/publish.sh" --mode pr --title "<title>" --body-file "$BODY"
rm -f "$BODY"
```

publish.sh の挙動と引数:

- `--mode push|pr`（必須）。`--title`・`--body-file` は `pr` で必須。`--base`・`--remote` は任意（省略時は自動検出。base はデフォルトブランチ、remote は 1 つなら自動 / 複数なら `origin`）。
- push: upstream があれば `git push`、無ければ `git push -u <remote> <branch>`。**`--force` 系は行わない。** push 失敗（非 fast-forward 等）は出力をそのまま提示し、`git pull --rebase` 等の方針はユーザーに判断してもらう。
- PR: `gh repo view` で GitHub 解決可否を判定（URL 文字列一致では判定しない）。base 自動検出、既存 open PR があれば新規作成せず URL を案内、`--draft` なし（ready for review）、`gh pr create --web` は使わない。作成成功後は PR URL をブラウザで開く。
- 権限不足時のアカウント切替: 有効アカウントが READ でも、write 権限を持つ別のログイン済みアカウントへ**自動で一時切替**して作成し、作成後は必ず元の有効アカウントへ戻す（**ユーザー承認不要**、切替→作成→復帰はスクリプト内で単一シェルとしてアトミックに実行され、復帰は無条件）。どのアカウントにも write 権限が無いときのみ PR を断念し、Web リンクでの作成を案内する。

#### 6-3. 結果報告

publish.sh の出力をそのまま要約して報告する:

- `PUSHED: <remote>/<branch>` … push 完了。
- `PR_CREATED: <url>`（および `PR_ACCOUNT: created=... restored=...` があればそれも）… PR 作成完了。使用/復帰アカウントも報告する。
- `EXISTING_PR: <url>` … 既存 PR があったため新規作成せず案内。
- 非ゼロ終了時はエラー出力をそのまま提示し原因を説明する。**push までを「成功」と偽らない**。とくに PR 断念（GitHub でない / write 権限なし）の場合は、push までは完了している旨と Web 作成の案内を明示する。

## 補足: 既存メモリ・設定との連携

- ユーザー実名・業務メールアドレスがメモリやコンテキスト経由で既知の場合、それらを **検出対象** として扱う（漏えいリスクが高いため）。scan の正規表現では拾えないので LLM が差分本文で確認する。
- `~/.claude/CLAUDE.md` 系で公開を許容している情報（GitHub ハンドル等）は警告レベルを下げてよいが、念のため列挙する。
- dotfiles リポジトリの場合は特に「業界・業種を特定できる内容」（C）に注意する。ホームディレクトリの設定ファイルは個人の業務文脈を反映しやすい。

## このスキルが目指さないこと

- 脆弱性スキャン（SQL injection、XSS、認証バイパス等）は対象外。これは公式の `security-review` スキルの担当。
- 機械的な静的解析ツール（`gitleaks`、`trufflehog` 等）の代替ではなく、人間と LLM の目視レビューを補助する位置付け。CI でツールを併用するのが望ましい。
- コードの品質・設計レビューは行わない。これは `pre-code-review` スキルの担当。
