# pre-commit-leak-check

ユーザーが Claude に commit / push / PR 作成操作を依頼したら、実行する前にステージ済み差分をレビューし、機密情報・個人情報・業界特定情報・端末固有情報が混入していないかを確認する。**コミット対象のファイル一覧と各ファイルの4観点チェック結果を表にまとめて提示し、ユーザーが明示的に承認するまで commit しない。** 承認を得たら commit し、続けて push する（upstream 未設定なら `git push -u` でローカルブランチを upstream に設定する）。承認時に**どこまで実行するか（push まで / PR 作成まで）をユーザーが選び**、`PR 作成まで` が選ばれた場合は push 後に PR を作成する。

機密情報の検出が 0 件であっても、承認ゲートは必ず通す。理由は次のとおり。チェックが「機密的に OK」でも、ユーザーがそもそもコミットしたくないファイル（誤って `git add` した実験ファイル・一時ファイル・無関係な変更）が紛れている可能性があり、それを人間の目で止められるようにするためである。表でファイル単位の全体像を見せることで、機密の有無とは独立に「このファイルは入れたくない」という判断をユーザーができる。

## なぜこのチェックが必要か

dotfiles や設定ファイルは公開リポジトリに置かれることが多く、過去にも「コメントの例示を汎用的なパス表記に修正」のような後追い修正が発生している。業務リポジトリでも、ローカル絶対パス・社内 URL・実名メールアドレス等が誤って混入すると、PR レビュー時の指摘やリビジョン履歴の汚染につながる。コミット直前に短時間で観点別レビューを挟むことで、後追い修正の発生を抑えられる。

## スキップ条件

ユーザーが明示的に以下のような指示を出した場合に限り、本スキルをスキップして直接 commit に進んでよい。

- 「チェック不要でコミットして」
- 「漏えいチェックなしで」
- 「機密チェックスキップ」

それ以外（単に「コミットして」と言われたケース）はすべて実施する。

## 実行フロー

### STEP 1: ステージ状況の確認

```bash
git status --short
git diff --staged
```

- ステージ済み差分が空の場合は、ユーザーに「ステージされている変更がありません。`git add` で対象ファイルを指定してください」と伝えて中断する。
- ステージ漏れがありそうな場合（`git status` に未ステージ変更がある）は、commit に含めるか確認する。
- バイナリファイルが含まれる場合は `git diff --staged --stat` で大きさを併せて確認し、本当に意図したものか確認する。

### STEP 2: 4 観点でのレビュー

差分テキストを読み、A〜D の 4 カテゴリそれぞれの観点で問題箇所を列挙する。観点ごとに優先度を意識し、リスクの高いもの（API キー・秘密鍵・実名/業務メール）は最優先で抽出する。

検出を機械的に補助するため、差分テキストに対して `grep` で以下のパターンも併せて確認する（行番号は実ファイルから取得する：`git diff --staged -U0 -- <file>` または `grep -n` を活用）。

```bash
# 端末固有パス
git diff --staged | grep -nE "/Users/[^/[:space:]]+|/home/[^/[:space:]]+"

# よくあるシークレットの前置詞
git diff --staged | grep -niE "api[_-]?key|secret|token|password|bearer|authorization"

# IP / ホスト名らしき文字列
git diff --staged | grep -nE "([0-9]{1,3}\.){3}[0-9]{1,3}"

# メールアドレス
git diff --staged | grep -nE "[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}"
```

`grep` の結果は補助情報であり、見落とし防止のために必ず差分テキスト全体も目視で読む。

#### A. 企業の機密情報

- 実在する企業名・プロダクト名（特に未公開のもの、社内コードネーム）
- 社内システム名・内部ツール名・社内ドメイン
- 顧客名・取引先名
- 内部 URL（社内 Confluence / Jira / Slack / Grafana など）
- 未公開の財務情報・KPI 数値・売上規模

#### B. 個人を特定できる情報

- 実名のフルネーム（自分・同僚問わず）
- 業務用メールアドレス（ドメインから業界が推測されるものを含む）
- 個人メールアドレス、電話番号、SNS アカウント、社員 ID
- 「user's email/name」のようにユーザー情報を保持しているメモリやコンテキストが差分に貼り付いていないか

ユーザー本人の実名や業務メールでも、公開リポジトリに残れば業界が特定されるリスクがある。検出したらフラグを立てる。

#### C. 業界・業種を特定できる内容

- 特定業界の専門用語の組み合わせ（例：金融商品名・医療コード・防衛関連略語・暗号資産プロダクト名・SaaS 製品名など）
- 業界固有の規制名・サービス名・商品名
- 業界文脈を含むサンプルデータ・コメント（テーブル名・カラム名・列挙値）

単独では問題ない単語でも、組み合わせで業界推定が容易になる場合は指摘する。

#### D. 端末・環境固有情報

- 絶対パス（`/Users/<実名>/...`、`/home/<実名>/...`、ホスト名を含むパス）
- ホスト名・ローカル IP・社内ネットワークアドレス
- API キー・アクセストークン・OAuth トークン・秘密鍵（`-----BEGIN ... PRIVATE KEY-----`、`AKIA...`、`ghp_...`、`xoxb-...` など）
- `.env` の生値・データベース接続文字列・パスワード
- シェル履歴・ローカル設定ファイル（macOS の `.DS_Store` など）

API キー・秘密鍵・トークン類は **検出時点で commit を止め、修正後の再チェックを必須とする**。漏えいすると即時のローテーションが必要になるため、リスクが他カテゴリと桁違いに高い。

### STEP 3: コミット対象とチェック結果の提示

**検出の有無にかかわらず、まず必ず以下の表を提示する。** 表にはステージ済みファイルを **1 件残らず** 列挙する（機密検出が無いファイルも省略しない）。これは、機密の有無とは別に「そもそもコミットしたくないファイルが混ざっていないか」をユーザーがファイル単位で確認できるようにするためである。

ファイル一覧は `git diff --staged --stat` から取得する。

```
## コミット対象とチェック結果

| パス | 変更概要 | A 企業機密 | B 個人情報 | C 業界特定 | D 端末固有 | 総合 |
|------|---------|-----------|-----------|-----------|-----------|------|
| path/to/file1 | 設定追加 (+12/-0) | OK | OK | OK | OK | ✅ OK |
| path/to/file2 | パス修正 (+3/-3) | OK | OK | OK | ⚠️検出 | ⚠️ 要確認 |
```

表の書き方:

- **「変更概要」** は差分の要旨と増減行数（`+追加/-削除`）を簡潔に。
- 各観点セルは `OK` か `⚠️検出` のいずれか。1 つでも `⚠️検出` がある行は総合を `⚠️ 要確認`、すべて OK の行は `✅ OK`。
- バイナリ・大容量ファイルは「変更概要」にサイズを併記し、意図したものか目立たせる。

表の直後に、次の確認文を **必ず** 添える:

```
👆 この中にコミットしたくないファイルが含まれていないか確認してください。
不要なファイルがあれば指定してください。`git restore --staged <file>` で個別に unstage します。
```

機密検出が 1 件以上ある場合は、表の **後に** 続けて下記の詳細フォーマットで該当項目を列挙する（検出 0 件なら詳細パートは不要で、そのまま STEP 4 の承認へ進む）。

以下のフォーマットで列挙する。

```
機密性チェック結果: 検出 N 件

## 高リスク（即時対応推奨）
※ API キー・秘密鍵・トークンは漏えいすると即時ローテーションが必要なため、ここに最優先で列挙する。

### [D] path/to/file:行番号
**検出**: `AKIA****************`
**リスク**: AWS アクセスキーがコミットに混入。漏えい時はキーローテーションが必要。
**修正案**: 該当行を削除し、環境変数 `AWS_ACCESS_KEY_ID` から参照する形に変更。

## カテゴリ別

### [A] 企業の機密情報
- path/to/file:12 `<社内ツール名>` → 「内部ツール」など汎用表記に置換
- ...

### [B] 個人を特定できる情報
- path/to/file:5 `user.name@example.com` → コミット用メール（`noreply@...`）または `user@example.com` に置換
- ...

### [C] 業界・業種を特定できる内容
- ...

### [D] 端末・環境固有情報
- path/to/file:3 `/Users/<your-username>/work/...` → `~/work/...` または `<project-root>/...` に置換
- ...
```

記述のルール:

- **検出箇所はファイル名と実ファイルの行番号で示す**（diff 上の行ではない）。行番号が特定できない場合は `-` と書く。
- **該当行の抜粋は最大 80 字程度**にトリミングし、長文を貼らない。秘密鍵などは全文を貼らずマスクする（`AKIA****************`）。
- **修正案は具体的に書く**。「適切に直してください」のような曖昧表現は避け、置換後の文字列例まで示す。
- **「削除推奨」「マスク推奨」「汎用化推奨」など処置タイプを明示**する。
- pre-code-review スキルの言い回しに揃え、「〜できないでしょうか？」「〜はいかがでしょうか？」を使い分ける（必須修正と推奨修正で語尾を変える）。

### STEP 4: ユーザー承認の確認（全ケース必須）

**この承認ゲートは検出 0 件でも必ず通す。承認を得るまで commit / push / PR 作成は実行しない。** これがこのスキルの中核となる不変条件である。表とコミットメッセージ案を提示したうえで、`AskUserQuestion` ツール（または同等の選択肢提示）で次のアクションを選んでもらう。

このスキルでは **どこまで作業するか（push まで / PR 作成まで）も承認時にユーザーが選ぶ**。リポジトリや作業内容によって PR まで作りたいか push で止めたいかは変わるため、範囲のどちらにも「推奨」を付けず、毎回フラットに選んでもらう。

承認を求める際は、コミットメッセージ案も併せて提示する:

```
コミットメッセージ案:
{type}: {内容}

承認後、このメッセージで commit し、選ばれた範囲まで実行します
（upstream 未設定のブランチは `git push -u origin <branch>` で upstream を設定）。
```

選択肢（状況に応じて提示する）:

1. **承認 — commit → push → PR 作成**: 表のとおりで問題なく、PR まで作りたい場合。commit→push 後に PR を作成し、ブラウザで開く。**この選択をもって PR 作成の承認とみなし、作成直前の追加承認は挟まない**（base / タイトル / 本文は自動生成する。内容はブラウザで確認・編集できる）。
2. **承認 — commit → push（PR は作らない）**: push までで止めたい場合。従来どおり commit→push まで実行する。
3. **ファイルを除外 / 検出箇所を修正してやり直す**: 表にコミットしたくないファイルがある、または機密検出を直したい場合。STEP 5 で個別 unstage / Edit→再ステージ→再チェックを行い、表を再提示する（除外と修正は併用可）。
4. **中止**: 方針を見直したい場合。スキルを終了する。

PR 肢（選択肢 1）の出し分け:

- 選択肢 1 は **PR を作成できる環境のときだけ提示する**。判定は `gh repo view --json nameWithOwner` が成功するか（gh が対象 remote を GitHub リポジトリとして解決できるか）で行う。**URL に `github.com` を含むかの文字列一致では判定しない**——`git@github-xxx:owner/repo.git` のような SSH ホストエイリアスを取りこぼすため。`gh` 未認証や GitHub 以外で失敗するなら選択肢 1 を出さない。
- **書き込み権限は STEP 6-5 で確認・調整する。** 有効アカウントが READ でも、write 権限を持つ別のログイン済みアカウントへ自動で一時切替して作成しうるため、ここでは権限までは判定しない（どのアカウントにも write 権限が無い場合のみ 6-5 で PR を断念する）。
- 選択肢 1 を出さない場合は「この remote では PR を作成できないため、実行は push までになります」と一言添える。満たさない環境で勝手に push までへ切り替えて黙って進めることはせず、範囲が push までになる旨を明示する。

注意:

- 機密検出（特に API キー・秘密鍵・トークンなどの高リスク）がある状態で選択肢 1 または 2（push を伴う実行）を選ぼうとしたら、「これは公開しても問題ない値ですか？」と確認し、そのまま commit してよいかを念押しする。高リスク検出はローテーションが必要になるため、原則は選択肢 3 で修正してもらう。
- 選択肢 3 では不要ファイルの除外と機密修正を併用しうる。その場合は両方こなしてから STEP 4 をやり直す。

### STEP 5: 除外・修正と再チェック

選択肢 3 を選ばれたら、まず「ファイルを除外するか / 検出箇所を修正するか（両方可）」を確認し、該当する手順を行う。

**ファイルを除外する場合**:

1. ユーザーが指定したファイルを **個別に** `git restore --staged <file>` で unstage する。`git reset`（全解除）は他の意図したステージを巻き込むため使わない。
2. unstage したファイルはコミット対象から外れる（作業ツリーの変更自体は残る）。
3. STEP 3 に戻り、表を再生成して再提示する → STEP 4 で改めて承認を取る。

**機密検出を修正する場合**:

1. 指摘箇所を Edit ツールで修正する。複数ファイルにまたがる場合は変更ごとに分けて Edit する。
2. 修正対象ファイルを **個別に** `git add <file>` で再ステージする。`git add -A` / `git add .` は意図しないファイルを巻き込むため使わない。
3. 再度 `git diff --staged` を読み、修正が反映されているか・新たな問題が混入していないか確認する。
4. STEP 3 に戻り、表を再生成して再提示する → STEP 4 で改めて承認を取る。残存問題があればこのループを繰り返す。

いずれの場合も、**承認（選択肢 1 または 2）が得られて初めて STEP 6 に進む**。

### STEP 6: commit 実行 → push →（任意で）PR 作成

承認（選択肢 1 または 2）が得られたら、commit してから、選ばれた範囲（push まで / PR 作成まで）を実施する。

#### 6-1. commit

システムプロンプトの git commit 規約に従って commit する。要点のみ再掲：

- HEREDOC でメッセージを渡す。
- `Co-Authored-By` 行を末尾に追加する。
- `--amend` は使わず、新規 commit を作る（pre-commit フック失敗時も amend ではなく原因修正→新規 commit）。
- `--no-verify` でフックをスキップしない。
- commit 後に `git status` で結果を確認する。

#### 6-2. push 先 remote の決定

```bash
git remote
```

- remote が **1 つだけ** → それを使う。
- remote が **複数** あり `origin` がある → 既定で `origin` を使う。
- remote が複数あり `origin` 以外も対象になりうる → どの remote に push するかユーザーに確認する。
- remote が **0 個** → push できない。「push 先 remote が設定されていません。`git remote add origin <URL>` で追加してください」と伝え、commit までで止める。

#### 6-3. upstream の有無を判定して push

現在のブランチに upstream（追跡先）が設定されているかを確認する。

```bash
git rev-parse --abbrev-ref --symbolic-full-name @{upstream}
```

- **成功（upstream あり）** → そのまま push する:

  ```bash
  git push
  ```

- **失敗（upstream なし）** → ローカルブランチを upstream に指定して push する:

  ```bash
  branch=$(git rev-parse --abbrev-ref HEAD)
  git push -u <remote> "$branch"
  ```

  `<remote>` は 6-2 で決めたもの（既定 `origin`）。`-u`（`--set-upstream`）により、以降このブランチは `git push` だけで追跡先に push できるようになる。

#### 6-4. push 結果報告

push 後、リモートブランチ名と（出力にあれば）リモート URL や PR 作成リンクを報告する。

push が失敗した場合（非 fast-forward など）は、エラー内容をそのまま提示し原因を説明する。**`--force` / `--force-with-lease` は本スキルでは行わない**。リモートを上書きするリスクがあるため、必要ならユーザーに `git pull --rebase` などの方針を判断してもらう。

**作業範囲が `push まで`（選択肢 2）ならここで完了。** `PR 作成まで`（選択肢 1）が選ばれている場合のみ 6-5 に進む。

#### 6-5. PR 作成（選択肢 1 のときのみ）

push が成功し、作業範囲が `PR 作成まで` の場合に実行する。

1. **前提の再確認（GitHub か）**: `gh repo view --json nameWithOwner` が成功することを確認する（gh が対象 remote を GitHub リポジトリとして解決できれば OK。SSH ホストエイリアスでも可。**URL の文字列一致では判定しない**）。失敗するなら PR を作成せず、理由（GitHub でない / `gh` 未認証）を明示して push までで完了報告する。黙って続行・回避しない。
2. **書き込み権限の確認とアカウント調整**: `gh repo view --json viewerPermission -q .viewerPermission` を確認する。

   - `WRITE` / `ADMIN` / `MAINTAIN` → 現在の有効アカウントで作成できる。そのまま次へ。
   - `READ` / `NONE`（権限不足）→ **write 権限を持つ別のログイン済みアカウントへ一時切替する（この切替はユーザー承認不要）**。手順:
     1. 現在の有効アカウント名を控える（戻す先）。`gh auth status --active` で取得する。
     2. `gh auth status` でログイン済みアカウントを列挙し、有効でないものを候補にする。`nameWithOwner` の owner と一致するアカウントを優先する。
     3. 候補へ順に `gh auth switch --user <候補>` で切替え、`gh repo view --json viewerPermission` が `WRITE`/`ADMIN`/`MAINTAIN` になる**最初のアカウント**を採用して以降を進める。
     4. **どのログイン済みアカウントにも write 権限が無い場合**は、元の有効アカウントへ戻したうえで PR を作成せず、その旨を明示して push 出力の Create PR URL（Web）での作成を案内する。push までを「成功」と偽らない。
   - **重要（アトミック実行）**: `gh auth switch` は **マシン全体のグローバル有効アカウント**を変更する（セッション限定ではない）。**別の Claude セッションや別ターミナルが同時にアカウントを切り替えうる**ため、`gh auth switch`（切替）→ `gh repo view`（権限再確認）→ `gh pr create`（作成）→ `gh auth switch`（復帰）は **必ず単一のシェル呼び出し（1 つの Bash 実行）内で連続して行う**。別々の呼び出しに分けると、間に有効アカウントが戻って作成が `must be a collaborator` で失敗しうる（各 Bash 呼び出しは新シェルで始まる点にも注意）。
   - **復帰は無条件**: PR 作成の成否にかかわらず、最後に必ず開始時の有効アカウントへ `gh auth switch` で戻す（`&&` で連結せず無条件に実行）。使用したアカウントと復帰先は結果報告に含める。

   アトミック実行の例（PR URL を捕捉し、復帰後に成功時のみブラウザで開く）:

   ```bash
   ORIG=$(gh auth status --active 2>/dev/null | grep "Logged in" | grep -oE "account [^ ]+" | awk '{print $2}')
   gh auth switch --user <write権限を持つアカウント>
   gh repo view --json viewerPermission -q .viewerPermission   # WRITE/ADMIN/MAINTAIN を確認
   PR_URL=$(gh pr create --base <base> --head "$branch" --title "<title>" --body-file - <<'BODY'
   <body>
   BODY
   )
   PR_EXIT=$?
   gh auth switch --user "$ORIG"            # 成否によらず無条件で復帰
   [ "$PR_EXIT" -eq 0 ] && open "$PR_URL"   # 成功時のみブラウザで開く（URL 直開きなので復帰後アカウントの read 権限に依存しない）
   ```
3. **対象ブランチの確認**: 現在の HEAD ブランチが base（デフォルトブランチ）と同一の場合、PR は作成できない。その旨を報告して終了する。
4. **既存 PR の確認**: 重複作成を防ぐため、既に open な PR がないか確認する。

   ```bash
   branch=$(git rev-parse --abbrev-ref HEAD)
   gh pr list --head "$branch" --state open --json url,number
   ```

   既存の open PR があれば新規作成せず、その URL を報告したうえで `open "<url>"` でブラウザでも開いて終了する（新規作成時と挙動を揃える）。
5. **base ブランチの決定**: デフォルトブランチを自動検出する。

   ```bash
   gh repo view --json defaultBranchRef -q .defaultBranchRef.name   # 例: main
   # gh で取れなければ git のリモート HEAD から:
   git symbolic-ref refs/remotes/origin/HEAD | sed 's#^refs/remotes/origin/##'
   ```

   いずれでも取得できない場合は base をユーザーに確認する。
6. **タイトル・本文の生成**:
   - **タイトル**: コミットが 1 件ならその subject をそのまま。複数なら変更内容を要約した 1 行にする。
   - **本文**: 変更概要（ファイル別の要点・背景）をまとめ、**末尾にシステムプロンプトの PR 規約フッター** を必ず付ける:

     ```
     🤖 Generated with [Claude Code](https://claude.com/claude-code)
     ```
7. **PR 作成とブラウザ表示**: STEP 4 で「PR 作成まで」が選ばれていること自体が PR 作成の承認なので、**作成直前の追加承認は行わず、そのまま作成する**（生成した base / タイトル / 本文は提示しない。内容はブラウザで確認・編集できる）。CLI で作成し（`--draft` は付けず ready for review。`gh pr create --web` は使わない——ブラウザ側で再入力させず CLI で内容を確定させるため）、本文は HEREDOC / 標準入力で渡す。作成成功後、`gh pr create` が出力した PR URL を `open` でブラウザに表示する（URL を直接開くため、ブラウザのログインセッションで表示でき、アカウント切替で復帰した有効アカウントの read 権限に依存しない）。

   ```bash
   PR_URL=$(gh pr create --base <base> --head "$branch" --title "<title>" --body-file - <<'EOF'
   <body>
   EOF
   ) && open "$PR_URL"   # 作成された PR をブラウザで開く（作成失敗時は open しない）
   ```

   ブラウザ表示は `open "$PR_URL"` の単一手段に統一する（フォールバック禁止の方針に沿い、`open` 失敗時に別手段へ切り替える分岐は設けない）。
8. **PR 結果報告**: 作成された PR の URL を報告する（権限不足でアカウント切替した場合は使用アカウントと復帰先も）。失敗した場合はエラー内容をそのまま提示し原因を説明する（push 同様、勝手なリトライ・回避はしない）。

## 補足: 既存メモリ・設定との連携

- ユーザー実名・業務メールアドレスがメモリやコンテキスト経由で既知の場合、それらをホワイトリスト的に **検出対象** として扱う（漏えいリスクが高いため）。
- `~/.claude/CLAUDE.md` 系で公開を許容している情報（GitHub ハンドル等）は検出しても警告レベルを下げてよいが、念のため列挙する。
- dotfiles リポジトリの場合は特に「業界・業種を特定できる内容」（C）に注意する。ホームディレクトリの設定ファイルは個人の業務文脈を反映しやすい。

## 補足: PR 作成の前提

- PR 作成（STEP 6-5）は **GitHub リポジトリ + `gh` CLI（認証済み）** が前提。GitHub かどうかは `gh repo view` の成否で判定する（**`github.com` の文字列一致では判定しない**——SSH ホストエイリアス `git@github-xxx:` を取りこぼすため）。満たさない環境では選択肢 1 を出さず、push までで明示的に停止する（黙って push までにフォールバックして進めることはしない）。
- 有効 gh アカウントが対象リポジトリに **write 権限を持たない**場合、write 権限を持つ別のログイン済みアカウントへ**自動で一時切替**して作成し、作成後は必ず元の有効アカウントへ戻す（**ユーザー承認不要**）。どのログイン済みアカウントにも write 権限が無いときのみ PR を作れず、Web リンクでの作成を案内する。`gh auth switch` はマシン全体のグローバル有効アカウントを変えるため、復帰は無条件で行い、使用/復帰アカウントを報告する。**切替→作成→復帰は単一シェル呼び出し内でアトミックに実行する**（別の Claude セッション/別ターミナルが有効アカウントを変えうるため。詳細は STEP 6-5-2）。
- base はデフォルトブランチを自動検出する。検出できない場合のみユーザーに確認する。
- 既に open な PR があるブランチでは新規作成せず、既存 PR の URL を案内する。
- タイトル・本文はコミットから自動生成する。**STEP 4 で「PR 作成まで」が選ばれていることをもって PR 作成の承認とし、作成直前の追加承認は挟まず** `gh pr create`（ready for review、`--draft` なし）で作成し、作成後は出力された URL を `open` でブラウザに表示する。内容はブラウザで確認・編集できる。

## このスキルが目指さないこと

- 脆弱性スキャン（SQL injection、XSS、認証バイパス等）は対象外。これは公式の `security-review` スキルの担当。
- 機械的な静的解析ツール（`gitleaks`、`trufflehog` 等）の代替ではなく、人間と LLM の目視レビューを補助する位置付け。CI でツールを併用するのが望ましい。
- コードの品質・設計レビューは行わない。これは `pre-code-review` スキルの担当。
