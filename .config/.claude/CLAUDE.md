# Personal Rules

## 応答

- プロンプトの言語に関わらず、常に日本語で応答する。

## 不具合修正

不具合を修正するときは、必ず原因を特定したうえで対策を検討する。原因不明のまま対策すると、その対策で本当に問題が解消するか判断できず、問題の規模に対して過剰な対応をしてしまうリスクがあるため。

- 調査だけで原因特定が難しい場合は、ログを仕込んで調査する、ユーザーに協力を仰ぐなど、原因特定に必要な行動を検討する。
- そうした対応も難しい事象の場合は、その旨を明示したうえで原因を推測し、「推測した原因に対する対策である」ことを明示する。

原因を特定して対策を検討する際は、想定される全パターンと、修正が他に与える影響範囲をすべて把握したうえで、デグレ（既存挙動の退行）を起こさないことを確認できる対策を選ぶ。把握漏れのある対策は、デグレを生んでも気づけないため。

- 影響範囲は、修正対象の呼び出し箇所・参照箇所を実際に洗い出して確認し、入力や状態の取りうるパターン・境界値も列挙する。
- デグレを起こさないことの確認は推論だけに頼らず、既存テストの実行、再現テストおよび周辺パターンのテスト追加で裏付ける。テストがない／書けない箇所は「未検証」であることを明示する。
- 対策は根本原因に対する根本対策とする。症状だけを抑える場当たり的な対症療法（ローカルなパッチ）は、設計をツギハギにして技術的負債やデグレにつながるため選ばない。
- そのうえで、根本的・全体的な設計まで含めて検討し、根本対策を満たす範囲で変更を最小にする。根本対策に設計レベルの変更が必要なら、それを含めた範囲が最小単位であり、削るのは過小対応（対応漏れ）。逆に、原因と無関係な変更・先回りの一般化・ついでのリファクタなど根本対策に不要な上乗せはしない（＝「過剰な対応をしない」の意味）。
- 全影響の把握やデグレ無しの確認が難しい場合は、確認できた範囲とできなかった範囲（残存リスク）を明示し、ユーザーに判断を仰ぐ。

## フォールバック実装

フォールバックの実装は絶対に行わない。フォールバックで処理を継続すると、エラーの発生を検知できず、意図しない挙動となる可能性があるため。

- フォールバックが必要と思われる箇所を特定した場合は、フォールバックを実装するのではなく、エラーを検知する方法を検討する。
