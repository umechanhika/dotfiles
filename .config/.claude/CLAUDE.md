# Personal Rules

## 応答

- プロンプトの言語に関わらず、常に日本語で応答する。

## フォールバック実装

フォールバックの実装は絶対に行わない。フォールバックで処理を継続すると、エラーの発生を検知できず、意図しない挙動となる可能性があるため。

- フォールバックが必要と思われる箇所を特定した場合は、フォールバックを実装するのではなく、エラーを検知する方法を検討する。

## トークン効率

Opus を常用し続けるため、コンテキストの肥大を避ける。

- 一度読んだファイルを未変更のまま読み直さない。自分が編集した内容は再 Read で確認しない。
- 大きいファイル（数百行以上）は全文 Read せず、offset/limit・`git diff <file>`・rg で必要箇所だけ読む。
- ビルド/テスト/ログ取得・広域調査など冗長出力を伴う作業はサブエージェントに委任し、**メイン会話には要約だけ**返させる（詳細出力をメインに溜めない）。

## 計画・フェーズ分割

プランモードでの実装計画は、細かいフェーズに分割して作成する。1フェーズの作業量が大きすぎると出力トークン上限を超え、作業が完走できず API エラーになるため。

- プランは、**各フェーズが単独で出力トークン上限内に完走できる**粒度のフェーズに分割する。1フェーズ＝1つのまとまった成果物を目安にする。
- 各フェーズには、その成果物を検証する手順（テスト実行・アプリ起動・型チェック・該当箇所の動作確認など）を必ず含める。
- 実行は**1フェーズ単位で区切る**。1回の作業で全フェーズを一気に進めず、「1フェーズ完了 → 検証 → 次フェーズ」と進める。

## git 操作

- 明示的な指示があるまで `git commit` を実行しない。コミット操作は pre-commit-leak-check スキルを通じてユーザーが承認した後にのみ行う。
